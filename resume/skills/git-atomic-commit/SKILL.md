## When to Use

- 작업이 완료되어 변경을 묶어 전달할 준비가 되었을 때

## Rules

- 변경 내용은 토픽(기능 구현, 버그 수정, 리팩토링, 설정 변경 등) 단위로 명확히 분리하여 원자적 커밋을 생성한다
- 각 커밋은 독립적으로 빌드 및 품질 검증이 가능해야 하며, 하나의 커밋이 하나의 논리적 변경 사항만 포함하도록 한다
- 커밋 메시지는 기존 저장소 관례를 따른다(제목 간결, 본문 상세, 언어는 기존 커밋에 맞춤)
- git 명령은 항상 `--no-pager`
- 커밋 단위로 원격 저장소에 푸시한다

## Merge Conflict Resolution

### Rules

- 충돌 시 더 현대적이고 우월한 코드 쪽을 선택
- 선택 근거는 `npm run check` 및 `npm run build` 통과로 검증

### Checklist

- 최신 의존성/패턴(예: stricter typing, 더 안전한 API)을 사용하는가
- 보안/로깅/에러 처리 정책을 더 잘 준수하는가
- 중복/복잡도를 더 줄이는가(DRY, SoC)
- 변경 검증으로 퇴행이 없는가(`skills/change-validation`)

## Checklist

- 포맷/린트/타입체크 등 품질 게이트가 통과했는가
- 커밋 범위가 하나의 목적을 가지는가
- 비밀/민감 정보가 포함되지 않았는가
