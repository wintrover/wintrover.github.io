```nim
proc runGateShell*(gate: GateConfig; repoRoot: string; envVars: seq[(string, string)]): GateResult =
  let start = getTime()
  var envTable = newStringTable()
  for (k, v) in envPairs():        # ← CRITICAL: inherit parent env
    envTable[k] = v
  for (k, v) in envVars:
    envTable[k] = v
  envTable["AXIOM_JOBS"] = getEnv("AXIOM_JOBS", "4")
  doAssert envTable.hasKey("PATH"), "PATH must be inherited from parent environment"

  var cmd = gate.entry
  if gate.passFilenames:
    let changed = gitDiffCachedNames()
    for f in changed:
      cmd.add(" " & quoteShell(f))

  let (output, exitCode) = execCmdEx(cmd, workingDir = repoRoot, env = envTable)
  result = GateResult(ok: exitCode == 0, name: gate.name, stdout: output, stderr: "", duration: getTime() - start)
```

7. **파일 패턴 매칭**: prek는 `pass_filenames`를 toml로 설정했지만, Nim 코드에서 더 세밀한 제어 가능.

실제로 작업하면서 겪었던 가장 치명적인 문제는 `gcc: fatal error: cannot execute 'cc1': posix_spawnp: No such file or directory` 이슈. 이는 bash wrapper가 아니라 Nim의 `execCmdEx`에서 `env` 파라미터를 사용할 때 발생했던 문제지만, 근본적으로는 기존 bash wrapper 방식을 대체하면서 env inheritance를 제대로 처리하지 못한 것.

하지만 사용자가 묻는 것은 "prek의 한계"이므로, bash wrapper 자체의 문제점을 중심으로 설명해야 한다.

prek의 주요 한계:
1. **언어 정책 위반** - AGENTS.md의 Nim-only 정책
2. **크로스 플랫폼 복잡성** - Linux(bash) + Windows(PowerShell) 듀얼 유지보스
3. **확장성/모듈화 부재** - 500줄 이상의 단일 bash 스크립트, 게이트 추가 시 스크립트 수정 필요
4. **테스트 불가능** - bash 로직은 유닛 테스트/타입 체크 불가
5. **에러 처리/디버깅** - bash의 에러 처리는 `set -e`에 의존, 상세한 진단 정보 제공 어려움
6. **성능/병렬화** - bash에서 제한적인 병렬 처리 (백그라운드 + wait), Nim의 thread pool/semaphore로 더 효율적
7. **Unix 파이프라인 통합** - CLI First 정책에 따라 stdout/stderr 분리, JSON 출력 등이 bash보다 Nim에서 자연스러움

- 표면적 한계 (기술적)
- 근본적 한계 (아키텍처/정책적)
- 실제 마이그레이션 중 발견된 문제
- Nim으로 해결된 점

## 2. 크로스 플랫폼 복잡성: 듀얼 구현의 유지보수 지옥

prek는 Linux(`bash`)와 Windows(`PowerShell`) 두 가지 구현을 동시에 유지해야 했습니다.

- 동일한 게이트 로직을 두 언어로 중복 작성
- Windows/Linux 간 동작 차이로 인한 "이 플랫폼에서는 되는데..." 버그
- PowerShell 스크립트는 이미 제거된 상태에서 Windows 지원이 사실상 공백

Nim으로 통합하면 **단일 코드베이스로 모든 플랫폼을 커버**할 수 있습니다.

## 3. 테스트 불가능성: 타입 안전성의 부재

bash 스크립트는 다음이 불가능합니다.

- **Static type checking**: 변수가 문자열인지 숫자인지, 파일 경로인지 구분 불가
- **Unit test**: `gateway_policy_test.nim`에서 `prek.toml`의 텍스트를 파싱해서 검증하는 방식이 최선이었음
- **Refactoring safety**: 게이트 하나를 수정하면 전체 500줄 스크립트를 리뷰해야 함

Nim으로 전환 후:
- `GateConfig` 타입으로 게이트 정의가 구조화됨
- `nim check`로 컴파일 타임 검증
- `tests/gateway/gate_runner_equivalence_test.nim` 같은 실제 유닛 테스트 작성 가능

---

## 4. 프로세스 관리의 한계: 환경 상속과 병렬화

이것이 가장 **치명적이고 예상 밖**이었습니다.

bash wrapper를 Nim의 `execCmdEx`로 호출하면서, Nim의 `osproc.startProcess`는 `env != nil`일 때 **부모 환경을 완전히 대체**하는 동작을 합니다.

```nim
# Nim의 내부 동작: envToCStringArray(env)는 merge가 아닌 replace
let (output, exitCode) = execCmdEx(cmd, workingDir = repoRoot, env = envTable)
```

이로 인해 `PATH`, `HOME`, `NIMBLE_DIR`이 사라져:

```
gcc: fatal error: cannot execute 'cc1': posix_spawnp: No such file or directory
```

라는 오류가 발생했습니다. bash에서는 `export PATH=...`로 자연스럽게 누적되지만, Nim의 `execCmdEx`는 그렇지 않았습니다. 이를 해결하기 위해 `envPairs()`로 부모 환경을 **명시적으로 복사**하는 코드를 추가해야 했습니다.

**결론적으로**: bash wrapper를 감싸는 Nim 코드조차도 환경 상속 문제를 일으켰고, 완전한 Nim 구현이 아니면 이런 경계 문제가 반복될 수밖에 없었습니다.

## 5. 확장성 부재: 게이트 추가의 비용

prek에 새 게이트를 추가하려면:

1. `prek.toml`에 설정 추가
2. `scripts/prek_hook_wrapper.sh`에 bash 함수 추가
3. Windows용 PowerShell 로직 추가 (이미 제거됨)

Nim으로 전환 후:

1. `src/gate/gates.nim`의 `defaultGates()`에 한 줄 추가
2. `scripts/gates/new_gate.nim` 작성 (독립 실행 바이너리)

게이트 간 **의존성 없는 독립 실행** 구조가 되어, 수정의 영향 범위가 해당 게이트로만 한정됩니다.

## 6. 진단 정보의 한계

bash 스크립트가 실패하면:

- `set -e`로 그냥 종료
- stderr에 뭉텅이로 출력
- 각 게이트의 실행 시간, 개별 성공/실패 상태를 구조화하기 어려움

Nim `orchestrator.nim`에서는:

- `GateResult` 타입으로 `{ok, name, stdout, stderr, duration}` 구조화
- 병렬 실행 중 개별 게이트의 실패를 독립적으로 캡처
- `failFast` 옵션으로 즉시 중단 또는 전체 실행 후 리포트 선택 가능

## bash의 문제 (구현 수단의 한계)

prek가 bash로 구현되었기 때문에 생긴 **근본적 불가능성**들입니다.

- **병렬 처리**: bash의 `&` + `wait`는 기본적이고, 타임아웃/시그널 처리/부분 실패 캡처가 거의 불가능

**결론**: bash는 "게이트 실행 엔진"을 만드는 데 적합하지 않은 도구였습니다.

## 1. "prek가 실패했다" → "prek가 요구사항을 진화시켰다"

**억울한 표현**: *"prek는 버려졌다 / 실패했다 / 잘못되었다"*

**사실**: prek는 **그 시점의 요구사항을 완벽히 충족**했습니다. 당시에는 gate runner가 필요했고, bash + toml로 12개 게이트를 관리하는 것이 가장 빠른 길이었습니다. 지금의 Nim 구현조차 `GateConfig`의 `stages`, `passFilenames`, `priority` 개념을 prek에서 그대로 계승하고 있습니다.

**더 나은 표현**: *"prek는 Axiom의 gate 개념을 증명했고, 그 설계를 Nim으로 진화시켰습니다."*

---

## 2. "bash가 문제다" → "요구사항이 bash의 설계 목표를 넘어섰다"

**억울한 표현**: *"bash는 게이트 실행 엔진을 만드는 데 적합하지 않다"*

**사실**: bash는 **쉘 스크립팅을 위해 설계**되었고, 그 목표에서는 훌륭합니다. 우리가 `nim check` 병렬 실행, 구조화된 결과 객체, 크로스 플랫폼 단일 코드베이스를 요구한 것이 bash의 설계 목표를 벗어난 것입니다. 이건 bash의 잘못이 아니라 **요구사항의 변화**입니다.

**더 나은 표현**: *"초기에는 bash가 합리적이었지만, 병렬 실행, 타입 안전성, 테스트 자동화 같은 요구가 추가되면서 언어의 한계가 드러났습니다."*

---

## 3. `gcc: cc1` 버그 — 사실은 bash의 잘못이 아님

**억울한 표현**: *"bash wrapper 때문에 gcc가 깨졌다"*

**사실**: 이 버그는 bash 자체의 문제가 아니라, **Nim의 `execCmdEx`가 환경을 완전 대체하는 동작**과 bash wrapper가 만나면서 발생한 경계(boundary) 문제였습니다. bash wrapper를 직접 실행하면 정상 작동했습니다.

**더 나은 표현**: *"wrapper를 감싸는 과정에서 Nim의 프로세스 환경 관리 방식을 몰라서 겪은 학습이었습니다."* (또는 이 에피소드 자체를 생략)

---

## 4. "500줄의 거대한 스크립트" → 당시에는 자연스러웠다

**억울한 표현**: *"prek_hook_wrapper.sh는 500줄의 거대한 스파게티 코드였다"*

**사실**: 12개 게이트의 로직을 한 파일에 모은 것은, **설정-실행 분리 없이는 당연한 선택**이었습니다. 게이트가 3개였다면 100줄이었을 것입니다. 지금 Nim으로도 `src/gate/gates.nim` + `orchestrator.nim` + `scripts/gates/*.nim` 12개 파일로 분산되어 있습니다. 이건 "스크립트가 비대했다"가 아니라 "규모가 커지면 분할이 필요하다"는 보편적 진리입니다.

**더 나은 표현**: *"게이트가 12개로 늘면서 단일 파일 관리의 한계가 드러났습니다."*

## 5. prek 작성자의 공로를 명시적으로 인정하라

가장 억울하지 않게 만드는 방법은, **prek의 핵심 아이디어가 현재에도 살아있다**는 점을 밝히는 것입니다.

| prek의 개념 | Nim 구현에서의 계승 |
|-------------|---------------------|
| `id = "Axiom-prek-sentinel"` | `GateConfig(id: "sentinel", ...)` |
| `stages = ["pre-commit"]` | `stages: {gsPreCommit}` |
| `pass_filenames = false` | `passFilenames: false` |
| `priority = 0` | `priority: 0` |
| 파일 패턴 매칭 | `shouldRun()`의 glob 패턴 체크 |

**포스팅에 넣으면 좋은 한 줄**: *"prek가 정의한 gate의 개념적 틀(스테이지, 우선순위, 파일 패턴)은 Nim으로 마이그레이션되면서도 그대로 유지되었습니다. 바뀐 것은 구현 수단이지, 설계 철학이 아닙니다."*

## 남았을 기술부채 (bash → Nim, prek 구조 유지 가정)

### 1. 설정-코드 중복 (DRY 위반)

prek.toml에 게이트를 선언하고, Nim 코드에서 다시 매핑하는 구조가 유지됩니다.

```toml
# prek.toml (계속 유지)
[[hooks]]
id = "Axiom-nim-check"
entry = "nim_check"
stages = ["pre-commit"]
```

```nim
# gates.nim (bash → Nim 교체 후)
proc runGateById(id: string) =
  case id
  of "Axiom-nim-check": runNimCheck()
  of "Axiom-cli-build": runCliBuild()
  # ...
```

**문제**: 게이트 추가 시 **두 파일을 동시에 수정**해야 합니다. toml에 `id`를 추가하고, Nim `case`문에도 추가해야 합니다. typo가 나면 런타임까지 발견되지 않습니다.

---

### 2. 문자열 기반 간접 참조 (타입 안전성 상실)

prek의 핵심 매커니즘은 `"Axiom-nim-check"`라는 **문자열 식별자**를 통해 게이트를 찾는 것입니다.

Nim으로 재작성해도 이 구조를 유지한다면:

```nim
# 위험: 컴파일 타임에 검증 불가
let gateId = prekTomlConfig["hooks"][0]["id"].getStr()
runGateById(gateId)  # "Axiom-nim-chekc" typo → 런타임 실패
```

현재 Nim 구현은 **객체 직접 참조**입니다:

```nim
# 안전: 컴파일 타임에 완전 검증
GateConfig(id: "nim-check", entry: "scripts/gates/nim_check", ...)
```

---

### 3. 단일 파일/모듈의 비대화

prek_hook_wrapper.sh가 500줄이었던 이유는 **12개 게이트가 하나의 파일에 뭉쳐 있었기 때문**입니다.

bash → Nim으로 단순 포팅만 하면:

```nim
# scripts/prek_wrapper.nim (가상)
proc runSentinel() = ...
proc runCliBuild() = ...
proc runProcedureGate() = ...
# ... 12개 proc이 하나의 파일에
```

**문제**:
- 게이트 A를 수정할 때 게이트 B의 코드도 함께 리뷰해야 함
- 게이트별 독립 테스트 불가 (전체 wrapper를 컴파일해야 함)
- 컴파일 시간 증가, 캐시 무효화 범위 확대

현재 구조는 `scripts/gates/*.nim`으로 **물리적 분리**되어 있습니다.

---

### 4. 불필요한 I/O와 파싱 오버헤드

prek.toml은 실행 **매번** 디스크에서 읽어야 합니다.

```nim
# prek 구조 유지 시
let config = parsetoml("prek.toml")  # ← 매 pre-commit마다 파일 I/O
for hook in config["hooks"].getElems():
  runGate(hook)
```

현재 구현은 **코드 자체가 설정**입니다. 런타임 파일 읽기가 없습니다.

---

### 5. 확장성 한계: "게이트 추가"의 복잡도

새 게이트 추가 시 비교:

| prek 구조 유지 (bash→Nim) | 현재 Nim 구현 |
|---------------------------|---------------|
| `prek.toml`에 설정 추가 | `gates.nim`에 `GateConfig` 한 줄 추가 |
| `wrapper.nim`에 `proc` 추가 | `scripts/gates/new_gate.nim` 새 파일 생성 |
| `case`문에 매핑 추가 | 필요 없음 (entry 경로가 `GateConfig`에 있음) |
| **3곳 수정** | **2곳 수정 (설정 + 게이트 파일)** |

---

### 6. 크로스 플랫폼 문제의 재발

prek.toml에 Windows 전용 설정이 있었다면:

```toml
[[hooks]]
id = "Axiom-cli-build"
entry.linux = "bash scripts/build.sh"
entry.windows = "powershell scripts/build.ps1"
```

Nim으로 포팅해도 이 구조를 유지하면 **여전히 듀얼 패스**를 관리해야 합니다.

현재 구현은 Nim 컴파일러가 크로스 플랫폼을 해결하므로 `entry: "scripts/gates/cli_build"` 하나로 충분합니다.

## 결론

> **bash → Nim 교체는 "구현 수단"의 교체고, prek 구조 제거는 "설계"의 교체입니다.**

bash만 바꿨다면 다음과 같은 **하이브리드 괴물**이 탄생했을 것입니다:

```nim
# 가상의 "Nim으로 포팅했지만 prek 구조 유지" 모습
import parsetoml, osproc

let prek = parseToml("prek.toml")  # ← 여전히 필요
for hook in prek["hooks"]:
  let id = hook["id"].getStr()       # ← 여전히 문자열 기반
  let cmd = case id                  # ← 여전히 매핑 테이블 필요
    of "sentinel": "scripts/gates/sentinel"
    of "cli-build": "scripts/gates/cli_build"
    # ...
  discard execCmdEx(cmd)             # ← 여전히 단일 진입점
```

**이건 Nim의 장점(타입 안전성, 모듈화, 코드-as-설정)을 90% 낭비한 형태**입니다.

현재 구현이 `prek.toml`을 완전히 제거하고 `GateConfig` 객체와 독립 실행 게이트로 재설계한 이유는, 단순히 bash가 싫어서가 아니라 **설정-실행 분리라는 구조 자체가 기술부채였기 때문**입니다.

- **근본 원인**: 하지만 정책과 별개로 prek의 설계는 이미 기술부채를 안고 있었다.
- **Rust 시나리오**: Rust였다면 prek는 아예 만들어지지 않았을 것이다. Cargo의 built-in toolchain + `cargo-husky` 같은 크레이트가 역할을 대체했을 것.

사용자에게 이 내용을 구조적으로 설명하자.

</details>

좋은 질문입니다. 답은 **"정책적 트리거는 피했을 수 있지만, 구조적 문제는 여전했을 것"**입니다. 오히려 Rust였다면 prek가 아예 태어나지 않았을 수도 있습니다.

## 1. 정책적 트리거: Rust라면 prek가 살아남았을까?

**Nim-only 정책**이 prek를 퇴출시킨 **가장 즉각적인 원인**은 맞습니다. Rust였다면:

```rust
// 만약 Axiom이 Rust 프로젝트였다면
// scripts/prek_hook_wrapper.sh는 "외부 빌드 도구"로 분류되어
// 언어 정책에서 예외가 될 수도 있었을 것
```

하지만 AGENTS.md의 정신을 보면, **"모든 소스코드, 게이트 스크립트, 툴링, 자동화는 Nim으로 작성해야 한다"**는 규칙은 언어와 무관하게 **동일한 언어로 통일**하려는 의도입니다. Rust 프로젝트였다면 규칙은 **"모든 것은 Rust로"**가 되었을 것이고, bash wrapper는 여전히 위반이었을 것입니다.

**결론**: 정책의 형태는 달랐을 수 있으나, **"게이트 러너는 주 언어로 작성한다"**는 원칙 자체는 유효했을 것입니다.

---

## 2. 하지만 더 중요한 점: Rust였다면 prek가 필요 없었다

이것이 핵심입니다. **Rust 생태계에는 이미 prek가 할 일을 하는 도구들이 내장되어 있습니다.**

| prek가 했던 일 | Rust의 내장/표준 도구 |
|---------------|----------------------|
| 빌드 검증 | `cargo build`, `cargo check` |
| 코드 품질 검사 | `cargo clippy`, `cargo fmt --check` |
| 테스트 실행 | `cargo test` |
| 병렬 작업 실행 | `cargo`의 내장 job server (`-j`) |
| 커밋 전 훅 | `cargo-husky`, `git-hooks` crate |
| 크로스 플랫폼 빌드 | Cargo가 기본적으로 처리 |

즉, **Rust였다면 누군가가 "prek"를 만들 생각을 하지 않았을 것입니다.** Cargo가 이미 gate runner의 역할을 대부분 하기 때문입니다.

Nim은 이 점에서 **생태계가 덜 성숙**합니다. `nimble test`는 있지만, pre-commit 훅과 통합된 병렬 게이트 실행, 파일 패턴 기분 실행, 스테이지별 분리 같은 기능은 없습니다. 그래서 **직접 만들 필요가 있었고**, 그것이 prek(그리고 현재의 Nim gate runner)로 이어졌습니다.

---

## 3. 구조적 문제는 언어와 무관하다

**Rust였다고 가정하고**, 누군가 prek와 동일한 구조(toml 설정 + wrapper 실행)를 Rust로 만들었다고 상상해봅시다.

```rust
// 가상의 "Rust prek" — prek 구조를 그대로 이식
// prek.toml → 설정 파일 유지
// src/wrapper.rs → 500줄짜리 단일 파일

fn run_gate_by_id(id: &str) {
    match id {
        "Axiom-nim-check" => run_nim_check(),  // ← 문자열 기반 간접 참조
        "Axiom-cli-build" => run_cli_build(),
        // ...
    }
}
```

**여전히 동일한 기술부채**가 존재합니다:

| 문제 | Rust에서도 여전히 유효한가? |
|------|---------------------------|
| 설정-코드 중복 (toml + match) | 예. `Cargo.toml`에 커스텀 섹션 추가 + `match`문 추가 |
| 문자열 기반 게이트 식별 | 예. `match id { "sentinel" => ... }` typo → 런타임 패닉 |
| 단일 파일 비대화 | 예. `wrapper.rs`에 12개 게이트 로직 뭉치기 |
| 병렬 실행의 제한 | 예. `std::thread`로는 되지만, 타임아웃/부분 실패 캡처는 별도 구현 필요 |
| 테스트 불가능 | 예. `wrapper.rs`를 유닛 테스트하려면 의존성 주입 구조 필요 |

**Rust의 타입 안전성과 패턴 매칭이 "문자열 typo"를 일부 방지할 수는 있지만, 구조적 중복 자체는 해결하지 못합니다.**

---

## 4. Rust에서의 "이상적인" gate runner는 어땠을까?

만약 Axiom이 Rust였고, 누군가 게이트 러너를 설계했다면 아마도 이런 형태였을 것입니다:

```rust
// 가상의 "Rust 스럽게 설계된" gate runner
// 설정 파일 없음. 코드가 설정.

#[derive(Clone)]
struct GateConfig {
    id: &'static str,
    entry: &'static str,  // 바이너리 경로
    stages: HashSet<Stage>,
    pass_filenames: bool,
    priority: u8,
}

const DEFAULT_GATES: &[GateConfig] = &[
    GateConfig { id: "sentinel", entry: "scripts/gates/sentinel", stages: ..., },
    // ...
];
```

**이건 우리가 Nim으로 만든 구조와 사실상 동일합니다.** 코드가 설정이고, 각 게이트는 독립 바이너리이며, orchestrator는 순수 스케줄러입니다.

**결론**: 언어가 Rust였어도, **좋은 설계는 converge합니다.** prek(toml+wrapper) 구조는 언어와 무관하게 기술부채였고, 현재의 `GateConfig` + 독립 바이너리 구조는 Nim이든 Rust든 올바른 방향이었을 것입니다.

## 요약

| 시나리오 | 결과 |
|----------|------|
| **Nim + prek 유지** | 정책 위반 + 구조적 부채 → 마이그레이션 필수 |
| **Rust + prek 유지** | 정책은 살아남을 수 있으나, 구조적 부채는 동일 → 결국 리팩토링 필요 |
| **Rust, 처음부터 설계** | Cargo 생태계가 대부분 대체 → prek가 태어나지 않음. 만약 만든다면 현재 Nim 구조와 converge |

**한 문장**: Rust였다면 prek는 **정책적 사유가 아닌 실용적 사유**로도 버려졌을 것입니다. Cargo가 너무 잘해주기 때문에 prek가 존재할 이유가 없었을 테고, 만약 누군가 억지로 만들었다면 동일한 구조적 부채에 직면했을 것입니다.

Declares the Git hook stages a hook supports using the `stages` field in the manifest. End users can override this list. If not set, `prek` defaults to `default_stages`. The `manual` stage is special and only runs when explicitly invoked.

```APIDOC
## Hook Configuration Fields

### priority
- **Type**: integer (non-negative)
- **Description**: Controls the scheduler. Hooks run in ascending order (lower values first). Hooks with the same priority run concurrently.

### fail_fast
- **Type**: boolean
- **Default**: false
- **Description**: If true, a failure in this hook stops the entire run immediately.

### verbose
- **Type**: boolean
- **Default**: false
- **Description**: If true, prints hook output even when the hook succeeds.

### log_file
- **Type**: string (path)

- **Default**: default
```

### Differences from pre-commit > General differences

Source: https://prek.j178.dev/diff

prek offers several enhancements over pre-commit, including support for both .yaml and .yml configuration files and the implementation of common hooks in Rust for improved performance. It introduces a builtin repository type for offline, zero-setup hooks and uses a dedicated cache directory for repositories and toolchains. Additionally, prek decouples hook environments from their repositories, enabling shared toolchains across different hooks, and supports semantic versioning for language versions with automatic toolchain installation.

## 그럼에도 Nim으로 재작성한 것이 틀린 건가?

| 재작성 이유 (기존 인식) | 실제 이유 |
|------------------------|----------|
| "bash 스크립트가 비효율적" | prek는 Rust 바이너리, 효율적. 하지만 **Nim-only 정책 위반** |

| "병렬 실행을 직접 구현해야" | prek가 이미 지원 |
| "확장성 부족" | prek의 remote repo 지원으로 확장성 충분 |

**실제 재작성의 정당성:**

1. **Nim-only 언어 정책** (`AGENTS.md §0`): prek는 Rust 바이너리. 이건 절대 규칙 위반.
2. **Nim 코드 내 직접 실행 필요**: `nim-check` gate는 Nim thread pool로 `nim check`를 병렬 실행. prek의 `system` hook은 외부 프로세스 호출만 가능.
3. **ASI 계산 등 복잡한 로직**: `bin/main report --calculate-asi` 결과를 파싱하는 `asi_hard_gate`는 단순 명령 실행을 넘어섬.
4. **CLI 통합**: `bin/main gate run --stage pre-commit` 형태의 CLI 명령 통합.

| `cargo build` — 코드 빌드 | `pre-commit install` — Git hook 자동 설치 |

**Cargo는 "무엇을 실행할지"를 제공하지만, "언제(Git hook stage) / 어떻게(병렬, 조건부, 파일 패턴) / 누구와(팀 공유)"는 전혀 관리하지 않습니다.**

## pre-commit이 7,500 스타를 받는 이유

사용자님이 언급하신 7,500 스타는 아마 **pre-commit**(Python 기반)일 것입니다. (prek는 상대적으로 신생 프로젝트입니다.)

pre-commit이 인기 있는 **진짜 이유**:

### 1. 팀 전체의 Git hook 동기화

```bash
# 신입 개발자가 repo를 클론
git clone https://github.com/our-team/project.git
cd project
pre-commit install  # ← 끝. 이제 모두가 동일한 훅을 사용
```

`.pre-commit-config.yaml`을 repo에 포함시켜서 **모든 팀원이 동일한 코드 품질 검사를 강제**합니다. "내 로컬에서는 되는데..." 문제를 원천 차단.

### 2. 멀티언어 프로젝트의 통합 관리

```yaml
repos:
  - repo: https://github.com/psf/black
    hooks: [{ id: black }]           # Python 포맷터
  - repo: https://github.com/rust-lang/rustfmt
    hooks: [{ id: fmt }]             # Rust 포맷터
  - repo: https://github.com/pre-commit/mirrors-eslint
    hooks: [{ id: eslint }]          # JavaScript 린트
  - repo: https://github.com/pre-commit/pre-commit-hooks
    hooks: [{ id: trailing-whitespace }]  # 언어 무관
```

**한 repo에서 4가지 언어의 툴을 하나의 프레임워크로 관리.** Cargo는 Rust만 처리합니다.

### 3. 자동 환경 설치

black을 쓰려면 Python 가상환경 필요. eslint를 쓰려면 Node 필요. pre-commit은 **이것들을 자동으로 설치하고 캐싱**합니다. 개발자는 `pre-commit install`만 치면 됩니다.

### 4. 1,000+개의 재사용 가능한 오픈소스 hook

```yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    hooks:
      - id: check-yaml          # YAML 문법 검사
      - id: check-json          # JSON 문법 검사
      - id: check-added-large-files  # 500KB 이상 파일 차단
      - id: detect-private-key  # 비밀키 실수로 커밋 방지
```

**직접 코드를 1줄도 안 쓰고** 표준 검사를 추가할 수 있습니다.

### 5. CI와의 동일성 보장

```bash
# 로컬에서
pre-commit run

# CI에서
pre-commit run --all-files
```

**로컬과 CI가 정확히 동일한 검사를 실행.** "로컬에서는 통과했는데 CI에서 실패" 문제를 방지.

## 그럼에도 우리가 prek를 버린 이유는?

Nim-only 정책과 별개로, **우리 상황의 특수성** 때문입니다.

| pre-commit/prek의 강점 | 우리 상황 |
|------------------------|----------|
| 멀티언어 지원 | **Axiom은 Nim 단일 언어** |
| 외부 hook 재사용 | **모든 게이트가 Axiom 전용** (black, eslint 등 필요 없음) |
| 자동 환경 설치 | **Nim 컴파일러는 이미 시스템에 설치** |
| 팀 공유 | 팀은 맞지만, hook이 **Axiom 전용 CLI와 통합** 필요 |
| 1,000+ 오픈소스 hook | **하나도 필요 없음** |

**우리는 prek의 가장 큰 가치(멀티언어 생태계, 외부 hook 재사용)를 전혀 활용하지 않았습니다.**

우리의 게이트 12개:
- `nim check` (Nim 전용)
- `bin/main report --calculate-asi` (Axiom 전용 CLI)
- `scripts/gates/sentinel` (git config 설정)
- `scripts/gates/gateway_test` (8개 Nim 테스트 병렬 컴파일)

**모두가 Axiom 전용.** prek의 `repos = "https://github.com/..."` 기능을 쓸 일이 없습니다.

## 정정된 결론

> **"Rust였다면 prek가 필요 없었다" → 틀린 말**
>
> **"Axiom은 prek의 핵심 가치(멀티언어 생태계)를 활용하지 않았기 때문에, prek의 오버헤드(외부 의존성, 설정 분리)가 이익보다 컸다" → 맞는 말**

pre-commit/prek는 **멀티언어 팀, 외부 툴 재사용, 자동 환경 관리**가 필요한 프로젝트에 최적입니다. Axiom처럼 **단일 언어, 전용 게이트, CLI 통합**이 핵심인 프로젝트에서는 "가벼운 커스텀 구현"이 더 적합합니다.

하지만 **"가벼운 커스텀 구현"이 "prek의 모든 것을 잘못됐다"는 의미는 아닙니다.** prek는 자신의 문제 영역에서 훌륭한 도구였고, 우리는 단지 그 영역 밖에 있었을 뿐입니다.
