## The Core Question

When building a verification engine for AI-generated code, architects face a fundamental dilemma: where do rules live, and who enforces them?

Axiom's answer is radical but necessary: **rules must precede code**.

## Rule-First Principle

Traditional software development follows an implicit pattern:

```
1. Write code
2. Discover bugs through testing
3. Add rules to prevent recurrence
4. Repeat
```

This reactive approach fails catastrophically when verifying AI-generated code. LLMs excel at crafting outputs that pass tests while concealing logical flaws—their probabilistic nature means bugs reappear in subtly different forms, bypassing test-based detection.

If rules aren't declared before code, a verification engine risks carving its own yardstick to fit AI-generated hallucinations—gradually lowering standards to accommodate probabilistic outputs.

Axiom inverts this paradigm:

```
1. Declare rules (invariants, contracts, safety properties)
2. Encode rules in verification artifacts
3. Prove code satisfies rules
4. Only then permit execution
```

This isn't just best practice—it's architectural necessity. A verification engine that discovers its own rules through code inspection would be circular: it would validate code against constraints derived from that same code, creating a self-justifying loop.

### What This Means in Practice

Every change to Axiom follows this sequence:

1. **Rule Declaration First**: When adding a new verification capability, the team first documents the formal rules in `CONTEXT.md` or `docs/spec/`
2. **Test Against Rules**: Tests validate that code correctly implements declared rules—not that code works in isolation
3. **Implementation Last**: Only after rules and tests stabilize does implementation begin

This isn't bureaucratic overhead. It's the only way to ensure the verification engine itself adheres to the same rigor it expects from verified code.

## Constitution-Ordinance-Annals Separation

Axiom organizes its rule hierarchy into three distinct layers, borrowing from legal systems that have refined this structure over centuries.

| Layer | Keyword | File Pattern | Purpose | Change Frequency |
|-------|---------|--------------|---------|------------------|
| **Constitution** | **Context** | `CONTEXT.md` | The immutable "Why" | Rare (project identity shifts only) |
| **Ordinances** | **Spec** | `docs/spec/*.md` | Detailed design specs | Moderate (algorithm/parameter changes) |
| **Annals** | **History** | `docs/history/*.md` | Decision records & context | Frequent (implementation logs) |

### Layer 1: Constitution (`Context`)

The constitution contains:

- **Project Purpose**: The fundamental "why" that guides all decisions
- **Core Architecture Principles**: Immutable constraints that define Axiom's identity
- **Trust Boundaries**: What the engine can and cannot assume
- **Quality Gates**: Pass/fail criteria that cannot be negotiated

```markdown
## 2. Core Architecture Principles (SSOT)

### Purity Hierarchy

- **Core Calculation Tier**: `src/core` computation uses `func` exclusively
- **Boundary Executor Tier**: `proc` exists only at OS/interfaces
- **Executor-only Rule**: Boundary `proc` cannot own business logic
```

This layer changes rarely—only when project identity itself shifts. Modifying `CONTEXT.md` requires architectural review because it affects every downstream decision.

### Layer 2: Ordinances (`Spec`)

Ordinances are detailed specifications:

Ordinances are detailed specifications:

- `bmc_core.md`: Model checking algorithms, Z3 interfaces, timeout rules
- `uap_logic.md`: Universal pipeline stages, counterexample promotion, invariant refinement
- `lean_proof.md`: Lean 4 semantics, tactic strategies, replay protocols

These files are loaded on demand—you read `CONTEXT.md` for direction, then dive into specific specs when implementing. This prevents the constitution from becoming cluttered with implementation details.

### Layer 3: Annals (`History`)

Annals capture decisions and changes:

```markdown
## 2026-03-26
- Core introduced Bottom-up Sorter, Vault Manager, Axiom Orchestration Loop
- Established stale axiom blocking and vault-based promotion paths
```

History stays separate from rules. When you need to understand *why* a rule exists, you consult the annals. But when you need to implement code, you focus on constitution and ordinances—without wading through historical noise.

### Why This Separation Matters

Without this structure, verification rules risk becoming entangled with implementation details and historical accidents. A developer might:

- Confuse "this is how we've always done it" (annals) with "this must never change" (constitution)
- Bury critical rules in changelog entries, making them invisible during code review
- Allow temporary compromises to become permanent architecture

Axiom's three-layer separation ensures:

- **Principles remain visible** (constitution)
- **Mechanics stay organized** (ordinances)
- **History stays searchable but separate** (annals)

## No-Downgrade Policy: The Integrity Hard Line

Axiom enforces one rule above all others: **integrity can only increase, never decrease**.

### The func→proc Trap

Nim's type system allows two function declarations:

```nim
# Pure: No side effects, no IO, no external state mutation
func computeValue(x: int): int {.noSideEffect.} =
  result = x * 2

# Impure: May perform IO, mutate state, throw exceptions
proc processValue(x: int): int =
  echo "processing"  # Side effect: IO
  result = x * 2
```

In Nim 2.0+, the `strictFuncs` mode and effect tracking make this distinction even more powerful:

```nim
# strictFuncs enforces purity at compile time
{.push strictFuncs.}
func verifyLogic(node: Node): Result[void, Error] {.raises: [], tags: [].} =
  # Compiler rejects any external state mutation or IO call
  if node.isValid: ok() else: err(InvalidNode)
{.pop.}
```

This is why Nim is optimized for Axiom's architecture: the language itself enforces purity boundaries at compile time, not just at code review.

During development, it's tempting to convert a `func` to `proc`:

> "I just need to add a quick debug log here. I'll change `func` to `proc`, add the log, and change it back later."

**This is forbidden in Axiom.**

Why? Because:

1. **Downgrade is a one-way door**: Once `proc` exists, developers will add more side effects. The mutation spreads.
2. **Purity is binary**: Either a function is pure or it isn't. "A little impure" is meaningless.
3. **Verification requires purity**: You cannot mathematically prove properties about functions with unpredictable side effects.

### Axiom's Enforcement Mechanism

Axiom maintains a **Purity Hierarchy**:

```
┌─────────────────────────────────────────────────────────────┐
│ Core Calculation Tier (src/core)                            │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ func-only zone: computation, parsing, normalization     │ │
│ │ Returns: Result[T], Option[T], VerifiedType[T]         │ │
│ │ Forbidden: IO, mutation, exceptions                     │ │
│ └─────────────────────────────────────────────────────────┘ │
│                                                             │
│ Boundary Executor Tier (src/ui, src/cli, gateway)          │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ proc-allowed zone: OS, TTY, filesystem, processes       │ │
│ │ Role: Execute core func results, inject dependencies    │ │
│ │ Forbidden: Business logic, domain rules                  │ │
│ └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

When a `proc` appears in `src/core`, it's flagged as **design debt**—a temporary compromise with a mandatory refactor plan. The code doesn't ship until:

1. The executor boundary is clearly marked
2. A "func promotion" task exists in the backlog
3. The reason for impurity is documented in `CONTEXT.md`

### What Happens When You Violate This Rule

In Axiom, downgrading `func` to `proc` isn't merely a code modification. It's treated as an act that destroys the system's mathematical integrity. Downgrades without proper justification (`CONTEXT.md` update) are blocked at the CI stage entirely.

Axiom's CI enforces this through the **Procedure Gate**:

1. If `src/core` files change, check for `proc` additions
2. If `func`→`proc` conversion appears in the diff, **fail the commit**
3. If `proc` exists without corresponding Boundary documentation, **fail the commit**

The only exception: `--style-only` changes (typos, comments, whitespace). Everything else requires architectural justification.

At compile time, Nim's effect system provides additional enforcement:

```nim
# [Layer 1: Core Calculation Tier]
{.push strictFuncs.}
func verifyNodeIntegrity(node: AxiomNode): Result[void, IntegrityError] {.raises: [].} =
  ## No IO, no mutation, no exceptions - mathematically pure
  if node.fingerprint == node.computedFingerprint():
    ok()
  else:
    err(IntegrityError.FingerprintMismatch)

# [Layer 2: Boundary Executor Tier]
proc executeVerification(node: AxiomNode, output: File) {.raises: [IOError].} =
  ## Thin executor: injects dependencies, delegates all logic to func
  let result = verifyNodeIntegrity(node)  # Calls pure function
  if result.isOk:
    output.write("Verification Passed\n")  # IO隔离在边界
  else:
    output.write("Verification Failed: " & $result.error & "\n")
{.pop.}
```

Notice the pattern: `verifyNodeIntegrity` is pure mathematics. `executeVerification` is a thin wrapper that handles IO. Business logic never crosses the boundary into impure territory.

### Real-World Example

When implementing report rendering, Axiom faced a choice:

```nim
# Initial tempting approach:
proc renderReport(data: seq[ProofResult]) =
  # Direct IO: writes to stdout
  for r in data:
    echo r.format()

# Enforced purity approach:
func renderReportLines(data: seq[ProofResult]): seq[string] {.noSideEffect.} =
  # Pure computation: returns strings
  for r in data:
    result.add r.format()

proc writeReport(data: seq[ProofResults]) =
  # Thin executor: injects output stream
  let lines = renderReportLines(data)
  for line in lines:
    echo line
```

The second approach separates concerns:

- `renderReportLines` is pure, testable, verifiable
- `writeReport` is a trivial executor that could be replaced with file output, network output, or test mock

This isn't over-engineering. It's **making verification possible**.

## Why These Principles Exist

Axiom verifies AI-generated code. To do this credibly, it must be **more rigorous than the code it verifies**.

Rule-First design ensures:

- Verification rules are never retrofitted to justify existing code
- Architecture principles are explicit, discoverable, and enforced
- Integrity only moves in one direction: upward

Constitution-Ordinance-Annals separation ensures:

- Principles stay visible without drowning in details
- Implementers find specifications easily
- History informs but doesn't clutter

The No-Downgrade Policy ensures:

- purity is preserved, not sacrificed for convenience
- Verification remains mathematically sound
- Technical debt is visible, tracked, and temporary

---

## Conclusion

These principles aren't aspirational—they're enforced by CI gates, code review, and architectural documentation. Every commit to Axiom must pass the Procedure Gate. Every `proc` in core must justify its existence. Every rule change must be documented before implementation.

> **The architecture must itself be verifiable.**

This is what it means to build a verification engine: **you cannot verify probabilistic code with probabilistic tools.** The foundation must be deterministic, mathematically proven, and architecturally pure.
