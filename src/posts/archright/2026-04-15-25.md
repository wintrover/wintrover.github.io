Two languages walk into a formal verification engine.

One brings a borrow checker, lifetimes, and `unsafe`.
The other brings `func`, `distinct`, and `{.raises: [].}`.

The first ensures the program **won't break**.
The second ensures the program **works as intended**.

This difference is not a theoretical distinction.
In an environment where AI generates code,
**semantic opacity is a bigger problem than memory safety.**

An LLM can generate code that compiles and passes tests,
while sufficiently hiding the author's intent.
Nim reduces that problem structurally.

<hr />

## Rust vs Nim: Not Performance, But "Semantic Interpretation Interface"

The usual comparison flows into benchmarks and FFI stories.
This is an approach that misses the core point.

- **Rust**: Machine trust optimization
- **Nim**: Human-AI semantic coherence optimization

Rust prevents memory corruption.
Nim prevents **intent corruption**.

Nim's core mechanisms:
- **Effect system** → Enforces the presence of side effects at compile time
- **distinct types** → Enforces the semantic boundary of a value
- **requires / ensures** → Specifies and verifies the legitimacy of transformations

These are not runtime checks.
They are enforced before the binary is generated.

### Summary of Core Differences

| Question | Rust | Nim |
|---|---|---|
| Does this crash? | Mostly no | Yes |
| Can you read the intent? | Depends on the author | Structurally revealed |
| Does AI code explain itself? | No | By default |

Rust marks "breaking points" with `unsafe`.
Nim makes "unbreakable areas" the default with `func`.

The question is this:

## Syntactic Simplicity: Where to Spend Attention

An LLM's attention is limited.

**Rust:**
```rust
fn explore<'a>(
    step: usize,
    trace: &'a BmcTrace,
    verdict: BmcVerdict,
    rule: &'a TransitionRule,
) -> BmcVerdict {
```
What many tokens convey here is:
- Lifetimes
- References
- Mutability
In other words, **memory management mechanisms**.

**Nim:**
```nim
func explore(step: int; trace: BmcTrace;
             verdict: BmcVerdict;
             rule: TransitionRule): BmcVerdict {.raises: [].} =
```
Every token here is:
- Domain variables
- Semantic constraints

→ Attention is focused on causality, not grammar.

<hr />

## Effect Enforcement: Making the Entire Code a Pure Zone

```nim
{.push raises: [], noSideEffect.}
# ...
{.pop.}
```
This block structurally enforces on a module level:
- No exceptions allowed
- No state mutations allowed

Rust does not have this level of structural purity enforcement.

<hr />

## Intent Visibility: "Why" is Revealed in the Declaration

```nim
func toSessionId*(raw: string): Result[SessionId, AxiomError] {.
  raises: [],
  requires: raw.len > 0,
  ensures: (not result.isOk()) or
           (sessionIdString(result.get()) == raw)
.}
```
This declaration specifies:
- Input conditions (`requires`)
- Output coherence (`ensures`)
- Failure model (`Result`)

This is not documentation.
It's a contract enforced by the compiler.

In Rust, the same "why" exists outside the code:
- Comments
- Documentation
- The author's mind

## func vs proc: Architecture Signals

- `func` → Pure function (Trusted zone)
- `proc` → Interacts with the outside world (Boundary zone)

In Axiom:
`proc` = Design debt signal

This distinction creates a **visible architecture** throughout the code.

<hr />

## distinct: Type = Semantic Boundary

```nim
type
  SessionId = distinct string
  OwnerId = distinct string
```
Effects:
- Type confusion → Blocked at compile time
- Refactoring → Structurally safe
- AI errors → Detected immediately

It's not just a simple type alias,
it's a **semantic unit separation device**.

## Structural Consistency: Problem → Code Mapping

Axiom's flow:
Problem → Domain Type → Implementation

- **Problem**: Race condition verification
- **Representation**: `BmcTrace`, `TransitionRule`
- **Implementation**: `func explore`

Each step preserves the structure of the previous step.
→ LLMs can understand it without interpretation.

### What Nim Provides

- No exceptions → `{.raises: [].}`

- Contracts → `requires` / `ensures`
- Semantic types → `distinct`
- Structure signals → `func` / `proc`
- Iteration removal → `macro`
- Compile-time verification → `static: doAssert`
