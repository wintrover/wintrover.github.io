When people hear ‚Äúformal verification‚ÄĚ, they imagine a single solver run that prints `Verified`.

That‚Äôs not what Axiom is optimizing for.

Axiom is built around one requirement: **if you say something is verified, you must be able to reproduce that claim deterministically**‚ÄĒincluding the exact query shape, the exact counterexample (if SAT), and the exact Lean replay (if promoted to safe).

## 1) BMC Core: Bounded Models, Not Hand-Wavy Guarantees

The BMC core is the ‚Äúboundary model‚ÄĚ layer of the engine: it takes a transition system, cuts it at a fixed bound, and asks the only question that matters inside that boundary:

> ‚ÄúDoes there exist a counterexample within N steps?‚ÄĚ

In Axiom‚Äôs reference race model, that boundary is explicit:

```nim
const
  bmcStepBound* = 10
```

That bound is not a random default‚ÄĒit‚Äôs the contract that defines what ‚Äúsafe‚ÄĚ even means in this mode. The semantics are also strict: a race is promoted only when the final state falls outside the set of sequential outcomes computed from the same `TransitionRule`.

The important architectural decision is that the *rule* and the *safety property* are not hardcoded into the solver path. Instead:

- External behavior is injected as a `TransitionRule(op, delta)` model.
- The SSOT of the safety property is `TransitionRule.invariant`.
- Encoding is performed as SMT-LIB2 (QF_LIA) and executed through Z3.

This is also where determinism stops being ‚Äúa preference‚ÄĚ and becomes ‚Äúmechanically enforced‚ÄĚ.

The Z3 query text is stamped with fixed-seed and single-thread constraints, and every invariant is asserted as a named clause so UNSAT cores are stable and actionable:

```lisp
(set-option :fixed_seed 17)
(set-option :smt.threads 1)
...
(assert (! (<= size 512) :named inv_bounded_growth.max_size))
...
(get-unsat-core)
```

In other words: the BMC core is not just ‚ÄúZ3 integration‚ÄĚ. It‚Äôs a deterministic boundary-model execution contract where solver outputs can be replayed and audited.

## 2) UAP Pipeline: Universal Automated Proof as an Orchestrated Repair Loop

BMC gives you a controlled counterexample search. But the real world is not a single bounded query; it‚Äôs a sequence:

1. Extract a model.
2. Find a counterexample.
3. Repair the implementation and/or strengthen invariants.
4. Re-prove.
5. Preserve only the minimum reusable evidence.

That sequence is what Axiom calls UAP: **Universal Acceptance Pipeline**.

UAP is not ‚Äújust‚ÄĚ automation. It has a strict output contract: one JSON result that includes translation, scans, repair rounds, proof artifacts, and promotion snapshots.

The core orchestrator is deliberately structured as a set of roles (Detective / Legislator / SWAT / Judge), because each stage produces different kinds of artifacts:

```text
AST + Source
  ‚Üí UniversalTranslation (AST ‚Üí IL/SMT mapping)
  ‚Üí initialScan (counterexample search + trace grounds)
  ‚Üí legislate (baseline invariant draft)
  ‚Üí executeRaid (repair + PoC + BDD generation hooks)
  ‚Üí certify (Lean replay + invariant finalization)
  ‚Üí investigate (final scan + verdict normalization)
  ‚Üí writeProofArtifact + writePromotionSnapshot
```

Two constraints matter here:

- ‚ÄúCounterexample not found‚ÄĚ is explicitly not equal to ‚Äúsafe‚ÄĚ.
- Safe promotion is prohibited unless Lean replay succeeds, even if SMT returns UNSAT.

So UAP is a self-healing pipeline, but it‚Äôs also an anti-vacuity pipeline: it refuses to treat solver silence as proof.

## 3) Lean 4 Reproducibility: Z3 Is the Oracle; Lean Is the Kernel

If SMT is the ‚Äúsearch engine‚ÄĚ, Lean 4 is the ‚Äúcourt of law‚ÄĚ.

The engine‚Äôs safety promotion rules explicitly treat Z3 UNSAT as a hint, not a conclusion. Final safe promotion requires a Lean-kernel-verifiable result (proof term or tactic replay).

This is why Axiom‚Äôs Lean side has hard constraints:

- Semantics are not allowed to hide inside axioms like `axiom transition`.
- Transition systems must be reconstructed through `def nextState` and `inductive Step`.
- Generated specs must be replayable under a pinned toolchain and stable tactic strategy.

Operationally, Lean replay is not a one-shot script. It‚Äôs a dynamic proof search loop driven by compiler feedback through Lean‚Äôs `--server` JSON-RPC path (LSP-style). The system treats ‚Äúproof search‚ÄĚ as an iterative, deterministic process: observe goals, propose tactics, replay, repeat.

The point is not that Lean always succeeds.

The point is that when it does succeed, it is **reproducible**‚ÄĒand when it fails, the failure itself is reproducible and promotable as actionable constraints.

## 4) Proof Vault: Fingerprint-Keyed Evidence Compression and Reuse

Most systems keep ‚Äúproof results‚ÄĚ as human-facing logs.

Axiom treats proofs as assets.

The Proof Vault is where those assets are stored under stable keys so they can be reused instead of re-derived. Two kinds of vaulting matter:

1) **Promoted invariants** (UAP-level): the engine preserves the latest `activeInvariant` and `refinedInvariant` as snapshots and auto-reinjects them into later runs for the same `modulePath/functionName`.

2) **Lean replay tactics** (kernel-level): successful tactics are stored keyed by theorem name plus a complexity fingerprint. On the next run, they become a warm-start strategy‚ÄĒpreferentially reusing the path with fewer attempts and shorter tactic sequences.

That ‚Äúfingerprint-first‚ÄĚ design is what turns the vault into a compression mechanism:

- You discard bulky intermediate artifacts (full ASTs, large traces) after success.
- You keep only minimal, fingerprint-addressable evidence that can deterministically recreate the proof attempt.

This is also why fingerprints are treated as first-class inputs across the pipeline (source checksum, SMT fingerprint, invariant fingerprint). If the fingerprint changes, Axiom does not pretend the old proof still applies‚ÄĒit demotes and schedules re-proof precisely where the change occurred.
