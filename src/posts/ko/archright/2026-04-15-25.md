두 언어가 정형 검증 엔진에 들어갔다.

하나는 대여 검사기, 라이프타임, `unsafe`를 들고 왔다.
다른 하나는 `func`, `distinct`, `{.raises: [].}`를 들고 왔다.

첫 번째는 프로그램이 **깨지지 않음**을 보장한다.
두 번째는 프로그램이 **의도대로 작동함**을 보장한다.

이 차이는 이론적 구분이 아니다.
AI가 코드를 생성하는 환경에서는,
**메모리 안전보다 의미 불투명성**이 더 큰 문제다.

LLM은 컴파일되고 테스트를 통과하면서도,
작성자의 의도를 숨긴 코드를 충분히 만들어낼 수 있다.
Nim은 그 문제를 구조적으로 줄인다.

<hr />

## Rust vs Nim: 성능이 아니라 “의미 해석 인터페이스”

일반적인 비교는 벤치마크와 FFI 이야기로 흐른다.
이건 핵심을 놓친 접근이다.

- **Rust**: 기계 신뢰 최적화
- **Nim**: 인간–AI 의미 정합성 최적화

Rust는 메모리 오염을 방지한다.
Nim은 **의도 오염**을 방지한다.

Nim의 핵심 메커니즘:
- **효과 시스템** → 부작용 존재 여부를 컴파일 타임에 강제
- **distinct 타입** → 값의 의미 경계를 강제
- **requires / ensures** → 변환의 정당성 명시 및 검증

이들은 런타임 검사가 아니다.
바이너리 생성 이전에 강제된다.

### 핵심 차이 요약

| 질문 | Rust | Nim |
|---|---|---|
| 이것이 충돌하는가? | 대부분 아니다 | 예 |
| 의도를 읽을 수 있는가? | 작성자에 의존 | 구조적으로 드러남 |
| AI 코드가 스스로 설명하는가? | 아니다 | 기본적으로 그렇다 |

Rust는 `unsafe`로 “깨지는 지점”을 표시한다.
Nim은 `func`로 “깨지지 않는 영역”을 기본값으로 만든다.

<hr />

## AI 시대의 진짜 질문

문제는 더 이상 이것이 아니다:
“이 코드가 안전한가?”

문제는 이것이다:
**“이 코드가 무엇을 의도했는지, 실행 전에 알 수 있는가?”**

<hr />

## 구문 단순성: 어텐션을 어디에 쓰게 할 것인가

LLM의 어텐션은 제한되어 있다.

**Rust:**
```rust
fn explore<'a>(
    step: usize,
    trace: &'a BmcTrace,
    verdict: BmcVerdict,
    rule: &'a TransitionRule,
) -> BmcVerdict {
```
여기서 많은 토큰이 전달하는 것은:
- 라이프타임
- 참조
- 가변성
즉, **메모리 관리 방식**이다.

**Nim:**
```nim
func explore(step: int; trace: BmcTrace;
             verdict: BmcVerdict;
             rule: TransitionRule): BmcVerdict {.raises: [].} =
```
여기서 모든 토큰은:
- 도메인 변수
- 의미 제약
이다.

→ 어텐션이 문법이 아니라 인과관계에 집중된다.

<hr />

## 효과 강제: 코드 전체를 순수 영역으로 만들기

```nim
{.push raises: [], noSideEffect.}
# ...
{.pop.}
```
이 블록은:
- 예외 금지
- 상태 변이 금지
를 모듈 단위로 강제한다.
Rust에는 이 수준의 구조적 순수성 강제가 없다.

<hr />

## 의도 가시성: “왜”가 선언에 드러난다

```nim
func toSessionId*(raw: string): Result[SessionId, AxiomError] {.
  raises: [],
  requires: raw.len > 0,
  ensures: (not result.isOk()) or
           (sessionIdString(result.get()) == raw)
.}
```
이 선언은 다음을 명시한다:
- 입력 조건 (`requires`)
- 출력 정합성 (`ensures`)
- 실패 모델 (`Result`)

이것은 문서가 아니다.
컴파일러가 강제하는 계약이다.

Rust에서는 동일한 “왜”가 코드 외부에 존재한다:
- 주석
- 문서
- 작성자의 머릿속

<hr />

## func vs proc: 아키텍처 신호

- `func` → 순수 함수 (신뢰 영역)
- `proc` → 외부 세계와 상호작용 (경계 영역)

Axiom에서는:
`proc` = 설계 부채 신호

이 구분이 코드 전반에서 **가시적 아키텍처**를 만든다.

<hr />

## distinct: 타입 = 의미 경계

```nim
type
  SessionId = distinct string
  OwnerId = distinct string
```
효과:
- 타입 혼동 → 컴파일 타임 차단
- 리팩토링 → 구조적으로 안전
- AI 오류 → 즉시 검출

단순한 타입 별칭이 아니라,
**의미 단위 분리 장치**다.

<hr />

## 구조적 일관성: 문제 → 코드 매핑

Axiom의 흐름:
문제 → 도메인 타입 → 구현

- **문제**: 레이스 조건 검증
- **표현**: `BmcTrace`, `TransitionRule`
- **구현**: `func explore`

각 단계가 이전 단계의 구조를 보존한다.
→ LLM이 해석 없이 이해 가능

<hr />

## 핵심 주장

대중적 프레이밍:
Rust = 안전 / Nim = 생산성

이건 틀렸다.
실제 구분은 이것이다:
**Rust → 안전**
**Nim → 정합성**

### 왜 지금 중요한가

메모리 안전은 이미 해결된 문제다.
하지만 이것은 해결되지 않았다:
**AI가 생성한 코드와 인간이 이해하는 의도가 일치하는가?**

이 격차는 점점 커지고 있다.

### Nim이 제공하는 것

- 예외 없음 → `{.raises: [].}`
- 순수성 → `{.noSideEffect.}`
- 계약 → `requires` / `ensures`
- 의미 타입 → `distinct`
- 구조 신호 → `func` / `proc`
- 반복 제거 → `macro`
- 컴파일 타임 검증 → `static: doAssert`

<hr />

## 결론

AI 보조 개발에서 기본값은:
안전이 아니다.
검증 가능성이다.

안전은 바닥이다.
정합성은 천장이다.

<hr />

### Axiom

Axiom은 Nim 기반 무결성 증명 엔진이다.
- BMC, k-귀납법
- SMT 해석
- Lean 4 증명 재생

코어는 전부:
`{.push raises: [], noSideEffect.}`
안에 존재한다.
→ 검증 엔진 전체가 순수 함수 집합

<hr />

### 마지막 질문

AI 생성 코드를 볼 때:
**“읽지 않고 왜를 알 수 있는가?”**

- 아니오 → 메커니즘이 의도를 가림
- 예 → 언어가 의도를 드러냄

이건 생산성 문제가 아니다.
정합성 문제다.
그리고 AI 시대에서,
정합성만이 지속 가능한 경쟁 우위다.
