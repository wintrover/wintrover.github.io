두 언어가 정형 검증 엔진에 들어갔다.

하나는 대여 검사기, 라이프타임, `unsafe` 블록을 들고 왔다. 다른 하나는 `func`, `distinct`, `{.raises: [].}`를 들고 왔다.

첫 번째는 프로그램이 세그폴트하지 않음을 보장한다. 두 번째는 프로그램이 **말하는 대로 작동함**을 보장한다.

이 구분은 학술적이지 않다. AI가 코드를 쓸 때, 메모리 안전은 의미 불투명성에 비하면 이미 해결된 문제다. LLM은 컴파일되고 `cargo test`를 통과하면서도 라이프타임 박사 학위 없이는 읽을 수 없는 의도를 인코딩하는 Rust 코드를 생성할 수 있다.

Nim은 그 문제가 없다. 이유를 설명하겠다.

## Rust가 아닌 Nim인 이유: 성능 비교가 아닌 의미 해석의 인터페이스

Rust vs. Nim 비교는 보통 벤치마크 차트와 FFI 워 스토리로 전락한다. 잘못된 축이다.

**Rust는 기계 신뢰를 최적화한다.** 대여 검사기는 프로그램이 메모리를 오염시키지 않음을 증명한다. `unsafe`는 그 증명이 깨지는 지점을 정확히 표시한다. 가치 있다. 하지만 불완전하다.

**Nim은 인간-AI 의미 정합성을 최적화한다.** 효과 시스템은 함수에 부작용이 없음을 증명한다. `distinct` 타입은 값이 게이트웨이 검증을 통과했음을 증명한다. `requires`/`ensures` 계약은 변환이 올바르게 왕복함을 증명한다. 이것들은 런타임 단정이 아니다. 컴파일러가 바이너리가 존재하기 전에 강제한다.

차이:

| 질문 | Rust의 답 | Nim의 답 |
|------|-----------|-----------|
| 이것이 충돌할까? | 대부분 아니오 | 예 |
| **의도**를 읽을 수 있는가? | 작성자에 따라 다름 | 키워드에 가시적 |
| AI 생성 코드가 스스로를 설명하는가? | 구조적으로 아니오 | 기본값으로 예 |

Rust는 안전하지 않은 코드를 **전염성** 있게 만든다 — `unsafe`를 호출하는 함수는 자신을 표시해야 한다. Nim은 부작용을 **구조적**으로 만든다 — 변환하거나 예외를 던지는 함수는 타입 서명에 선언해야 하며, 거짓이면 컴파일러가 거부한다.

AI 보조 개발의 질문은 "이것이 충돌할까?"가 아니다. "LLM이 500줄을 생성했을 때, 그 줄들이 **무엇을 의도했는지** 실행 전에 알 수 있는가?"다.

## 구문 단순성: LLM의 어텐션이 문법이 아닌 로직의 인과관계에 집중하도록 강제

LLM 어텐션은 유한하다. 보일러플레이트를 파싱하는 데 쓴 토큰 하나하나가 인과관계를 검증하는 데 쓰이지 않는 토큰이다.

언어 수준의 효과 강제를 보자:

```nim
{.push raises: [], noSideEffect.}

func explore(step: int; trace: BmcTrace;
             verdict: BmcVerdict;
             rule: TransitionRule): BmcVerdict {.raises: [].} =
  result = verdict
  if step == bmcStepBound:
    inc result.exploredPaths
    let finalX = trace.x[bmcStepBound]
    if bothWritesDone(trace, bmcStepBound):
      if not result.bothWritePathExists:
        result.bothWritePathExists = true
      if not isSequentialOutcome(finalX, result.sequentialResults,
                                  result.sequentialUniqueCount):
        if not result.raceDetected:
          result.raceDetected = true
          result.raceFinalState = finalX
          result.witness = trace
    return
  for owner in [ownerT1, ownerT2]:
    let next = applyTransition(trace, step, owner, rule)
    result = explore(step + 1, next, result, rule)

{.pop.}
```

`{.push raises: [], noSideEffect.}` 블록은 어떤 Rust 패턴도 할 수 없는 일을 한다: 전체 모듈에서 예외와 변이를 **구조적으로 금지**한다. 블록 내의 모든 `func`는 컴파일러에 의해 순수함이 검증된다. 라이프타임 없다. `&self` vs `&mut self` 없다. `RefCell`이나 `Arc<Mutex<>>` 없다.

LLM이 이 코드를 읽을 때 **전이 로직과 레이스 탐지**에 집중할 수 있다. 언어가 이미 순수 제약을 구문에 흡수했기 때문이다. 문법이 의미와 싸우지 않는다.

### 1. Rust가 강제하는 것

```rust
fn explore<'a>(
    step: usize,
    trace: &'a BmcTrace,
    verdict: BmcVerdict,
    rule: &'a TransitionRule,
) -> BmcVerdict {
    // 모든 참조에 라이프타임 주석
    // 모든 바인딩에 가변/불변
    // 이 함수가 순수함의 컴파일러 증명 없음
}
```

LLM의 어텐션 대역폭이 `'a`, `&'a`, `mut`에 소비된다. 이 토큰 중 어느 것도 **함수가 무엇을 하는지** 전달하지 않는다. **Rust가 메모리를 관리하는 방법**을 전달한다.

### 2. Nim이 집중하게 하는 것

```nim
func explore(step: int; trace: BmcTrace;
             verdict: BmcVerdict;
             rule: TransitionRule): BmcVerdict {.raises: [].} =
```

서명의 모든 토큰이 속성 이름이거나 도메인 타입이다. LLM의 어텐션은 `step`, `trace`, `verdict`, `rule` — 모델 체킹 알고리즘의 **인과 변수**에 향한다. 순수성과 예외 자유는 push pragma가 보증하며, 모든 호출 지점에서 소유권을 철자하는 것이 아니다.

이것이 **의미 초점의 강제 함수로서의 구문 단순성**이다.

## 의도 가시성: AI 생성 코드의 "왜 이렇게 작성되었는가"를 명확히 드러내는 Nim의 표현력

AI 생성 코드에서 가장 어려운 질문은 "이것이 무엇을 하는가?"가 아니다. "왜 이렇게 작성되었는가?"다.

Nim의 타입 시스템은 그 질문에 **선언에서** 답한다.

```nim
func toSessionId*(raw: string): Result[SessionId, AxiomError] {.
  raises: [],
  requires: raw.len > 0,
  ensures: (not result.isOk()) or
           (sessionIdString(result.get()) == raw)
.} =
  if raw.len == 0: return err(axiomErrEmptyString)
  if not isValidSessionPattern(raw):
    return err(axiomErrInvalidSession)
  ok(SessionId(raw))
```

이 서명을 소리 내어 읽어보라: "원시 문자열을 SessionId로 변환하는 함수는 예외를 던질 수 없고, 비어 있지 않은 입력을 요구하며, 실패하거나 결과가 올바르게 왕복함을 보장한다."

이것은 문서가 아니다. **컴파일러 강제 계약**이다. LLM이 이 패턴을 생성할 때:

1. `requires`는 전제조건을 노출한다 — AI가 이 입력이 유효한 조건에 대해 **생각해야** 했다.
2. `ensures`는 후제조건을 노출한다 — AI가 출력의 일관성을 **증명해야** 했다.
3. `Result[T, E]`는 실패 모드를 노출한다 — AI가 무엇이 잘못될 수 있는지 **명명해야** 했다.

Rust에서는:

```rust
fn to_session_id(raw: &str) -> Result<SessionId, AxiomError> {
    // 타입 시스템에 requires/ensures 없음
    // "왜"는 머릿속이나 주석에 있다
    if raw.is_empty() {
        return Err(AxiomError::EmptyString);
    }
    // ...
}
```

Rust 버전은 **무슨 일이 일어나는지** 알려준다. Nim 버전은 각 분기가 **왜 존재하는지** 알려주며, 컴파일러가 설명이 올바름을 검증한다.

### 3. 아키텍처 신호로서의 `func` vs `proc`

Axiom에서 `proc`은 기본값이 아니다. **설계 부채 신호**다.

```nim
{.push raises: [], noSideEffect.}
# ... 전체 코어 검증 엔진 ...
{.pop.}
```

코어의 모든 함수가 `func`이다. 효과 경계는 문서화되고, 강제되고, 완전하다. 코드베이스에서 `proc`를 보면 "이 함수는 외부 세계와 상호작용한다"는 뜻이다 — 그 정보가 **아키텍처적으로 가시적**이며, 구현 세부사항에 숨겨져 있지 않다.

CONTEXT.md는 `func`로 승격할 수 없는 이유와 함께 남은 모든 `proc`의 명시적 레지스트리를 유지한다. 이것은 주석 규약이 아니다. 빌드 시스템이 강제하는 **헌법 규칙**이다.

AI가 이 패러다임 내에서 코드를 생성할 때, 신뢰 영역에 부작용을 우연히 넣을 수 없다. 컴파일러가 헌법이며, `func`가 중요한 수정 조항이다.

## 구조적 일관성: 문제 → 표현 → 구현의 안정적 매핑

Nim이 AI-인간 협업에서 가장 깊은 장점은 단일 기능이 아니다. 문제 구조와 코드 구조 사이의 **안정적 매핑**이다.

Axiom 검증 파이프라인을 보자:

```
문제: "동시 상태 업데이트에서 레이스 컨디션이 발생할 수 없음을 검증하라"
  ↓
표현: BmcTrace, TransitionRule, BmcVerdict — 도메인 타입
  ↓
구현: func explore(...) — 순수 상태 공간 탐색
```

각 계층이 위 계층의 구조를 보존한다. 도메인 타입은 수학적 모델을 반영한다. 구현은 도메인 타입을 반영한다. LLM이 `BmcTrace`를 볼 때, 대여 검사기 장치를 풀지 않고도 변수가 나타내는 것을 이해할 수 있다.

### 4. 의미 경계로서의 distinct 타입

```nim
type
  SessionId* = distinct string
  OwnerId* = distinct string
  TrustedWordIndex* = distinct range[0..1023]
```

`distinct`는 미묘하지만 심오한 일을 한다: **모든 타입 경계를 의미 경계로 만든다**. 둘 다 `string`으로 백업되어도 `OwnerId`가 예상되는 곳에 `SessionId`를 실수로 전달할 수 없다. 타입 시스템이 이들이 **다른 의미**지 단순한 다른 이름이 아님을 강제한다.

AI 생성 코드의 의미:

- **환각된 타입 혼동이 컴파일 타임에 잡힌다.** LLM이 세션 식별자를 소유자 식별자로 사용하려고 하면 타입 검사기가 거부한다.
- **리팩토링이 구조적으로 안전하다.** `SessionId`의 검증을 변경해도 `OwnerId`에 영향을 주지 않는다. 이들은 별칭이 아니라 `distinct`이기 때문이다.
- **패턴 재사용이 명시적이다.** `makeGateway` 매크로가 모든 `distinct` 타입에 동일한 검증+승격 구조를 생성하여, AI가 한 번 학습하고 일관되게 적용할 수 있는 재사용 가능한 패턴을 만든다.

### 5. S-Expression DSL: 언어 내장 메타언어적 제어

```nim
func ite*(condExpr, thenExpr, elseExpr: SExpr): SExpr {.
  inline, raises: [].} =
  SExpr(kind: exprIte, text: "", args: @[condExpr, thenExpr, elseExpr])

func symInt*(name: string): SExpr {.inline, raises: [].} =
  SExpr(kind: exprSymbol, text: name, args: @[])
```

Axiom은 Nim에 내장된 S-expression DSL을 사용하여 SMT-LIB2 제약을 생성한다. 이 DSL은 **순수 함수적**(모든 생성자가 `func`)이고, **타입 안전**(SExpr은 완전히 매치된 `ExprKind`를 가진 ADT)하며, **의미론적으로 투명**하다 — `ite`는 SMT 로직에서 "if-then-else"를 의미하고, Nim 함수 이름이 정확히 SMT 개념이다.

LLM이 이 DSL을 통해 SMT 제약을 생성할 때, 수학적 의도(`∀ x, P(x) → Q(x)`)에서 코드(`forall(sym("x"), implies(p(sym("x")), q(sym("x"))))`)로의 매핑이 일대일이다. 보일러플레이트 계층 없다. 직렬화 의식 없다. 문제 구조가 **곧** 코드 구조다.

## 언어는 "성능"이 아니라 "정합성"의 문제

대중적 프레이밍은 이렇다: "안전을 위해 Rust, 생산성을 위해 Nim."

이 프레이밍은 진짜 트레이드오프를 놓친다.

**메모리 안전**은 2026년에 해결된 문제다. Rust가 해결했다. Go가 해결했다(대부분). 적절한 도구를 갖춘 Python도 규모에서 대부분의 메모리 문제를 잡을 수 있다.

**의미 정합성** — AI가 쓰는 것과 인간이 읽는 것이 동일한 의도를 나타내는지 — 은 해결되지 않았다. 사실, 모델이 구문적으로 유효하지만 의미론적으로 불투명한 코드를 더 많이 생성할수록 악화되고 있다.

Nim의 이 문제에 대한 장점:

1. **`{.raises: [].}`** — 컴파일러가 예외 자유를 증명한다. 숨겨진 `unwrap()` 패닉 없다.
2. **`{.noSideEffect.}`** — 컴파일러가 순수성을 증명한다. 계산 중 숨겨진 `println` 없다.
3. **`requires`/`ensures`** — Dr.Nim 계약이 전제조건과 후제조건을 타입 서명에 가시적으로 만든다.
4. **`distinct`** — 타입 바우하우스: 모든 타입 경계가 의미 경계다.
5. **`func` vs `proc`** — 키워드 구분이 아키텍처 의도를 한눈에 가시적으로 만든다.
6. **`macro`** — 메타프로그래밍이 검증된 패턴을 생성하여, 보일러플레이트와 LLM이 반복적인 검증 코드를 작성할 때 도입하는 "창의적" 오류 모두를 제거한다.
7. **`static: doAssert`** — 컴파일 타임 증명 검증. 증명이 성립하지 않으면 바이너리가 생성되지 않는다.

Rust에는 안전이 깨지는 곳을 표시하는 `unsafe`가 있다. Nim에는 안전이 **유지되는** 곳을 표시하는 `func`가 있다. 기본값은 신뢰다. 신뢰에서 벗어나려면 명시적 `proc` 선언과 아키텍처적 정당화가 필요하다.

AI 보조 개발에서 기본값은 **안전**이 아니라 **검증 가능**이어야 한다. 안전은 바닥이다. 정합성은 천장이다.

## Axiom

Axiom은 Nim으로 구축된 무결성 증명 엔진이다. 유계 모델 체킹, k-귀납법, SMT 해석, Lean 4 증명 재생을 사용하여 AI 생성 코드의 논리를 검증한다.

Axiom의 모든 코어 모듈은 `{.push raises: [], noSideEffect.}`로 시작하여 `{.pop.}`으로 끝난다. 검증 엔진은 컴파일러가 예외를 던지거나 외부 상태를 변이할 수 없음을 증명한 순수 함수 집합이다. 게이트웨이 타입은 Dr.Nim `requires`/`ensures` 계약을 가진 `makeGateway` 매크로로 생성된 `distinct` 타이다. SMT 계층은 순수 S-expression DSL이다. 증명은 `static: doAssert`로 컴파일 타임에 검증된다.

언어는 이 아키텍처를 가능하게 한 것이 아니라 **요구**했다. 컴파일러가 함수의 순수성을 증명할 때, 불순한 함수를 쓰기를 멈춘다. `distinct` 타입이 미검증 입력을 격리할 때, 원시 문자열이 도메인 로직에 들어오기를 멈춘다. `requires`/`ensures`가 추론을 노출할 때, 주석에 가정을 숨기기를 멈춘다.

Nim은 인간 의도와 AI 생성이 의미를 협상하는 인터페이스 계층이다. 빠르거나, 쉽거나, 패키지 관리자가 좋아서가 아니다. **구문이 곧 의미**이기 때문이다.

## 중요한 질문

AI 생성 코드를 리뷰할 때, 스스로에게 물어보라:

> "**어떻게** 읽지 않고 **왜**를 읽을 수 있는가?"

답이 라이프타임 주석, 소유권 이전, `impl` 블록을 파싱해야 한다면, 당신의 언어는 의도를 메커니즘 뒤에 숨기고 있다.

답이 키워드 — `func`, `distinct`, `requires`, `ensures`, `{.raises: [].}` — 에 있다면, 당신의 언어는 의도를 가시적으로 만들고 있다.

이것은 생산성 향상이 아니다. **정합성 향상**이다. 그리고 AI 생성 코드의 시대에, 정합성만이 의미 있는 유일한 경쟁 우위다.
