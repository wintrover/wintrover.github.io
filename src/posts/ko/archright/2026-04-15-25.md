두 언어를 정형 검증 엔진의 기반으로 두고 깊게 고민한 적이 있다.
하나는 대여 검사기와 라이프타임을 들고 온 Rust였고, 다른 하나는 `func`와 `distinct`를 앞세운 Nim이었다.

보통 이 두 언어를 비교할 때 사람들은 "Rust의 메모리 안전성 vs Nim의 C 호환성 및 생산성"을 이야기한다.
하지만 실제로 Axiom 같은 증명 엔진을 밑바닥부터 짜보면, 진짜 부딪히는 벽은 전혀 다른 곳에 있다.

특히 AI가 코드를 생성하는 환경에서, **가장 끔찍한 병목은 '컴파일 에러'가 아니라 '의미의 불투명성'이다.**
LLM은 테스트를 기가 막히게 통과하면서도, 작성자의 원래 의도를 교묘하게 비틀어버린 코드를 너무나 쉽게 만들어낸다.

## 완벽한 대칭은 없다: 메모리와 의도 사이의 트레이드오프

"Rust는 메모리 오염을 막고, Nim은 의도 오염을 막는다."
이렇게 딱 잘라 말하면 멋있어 보이지만, 현실의 엔지니어링은 그렇게 깔끔하게 떨어지지 않는다.

물론 Rust에서도 타입스테이트(Typestate) 패턴이나 뉴타입(Newtype) 패턴을 쓰면 도메인의 의미를 훌륭하게 강제할 수 있다. Rust가 의미 표현을 못 하는 언어라는 게 아니다. 문제는 **'그 의미를 표현하기 위해 치러야 하는 인지적 비용'**이다.

반대로 Nim이 무조건 정답인 것도 아니다. 매크로가 꼬이면 디버깅은 지옥이 되고, 생태계의 크기 차이는 때때로 큰 압박으로 다가온다. 이건 명백한 트레이드오프다.

하지만 내가 이 트레이드오프를 감수하고 Nim의 손을 들어준 이유는, AI와 협업하는 상황에서 코드를 읽고 검증하는 **'어텐션(Attention)'의 분배 방식** 때문이다.

<hr />

## 구문 단순성: 우리의 어텐션은 어디로 향하는가

LLM이 뱉어낸 코드를 리뷰할 때 우리의 인지 자원은 한정되어 있다.

**Rust 코드를 보자:**
```rust
fn explore<'a>(
    step: usize,
    trace: &'a BmcTrace,
    verdict: BmcVerdict,
    rule: &'a TransitionRule,
) -> BmcVerdict {
```
이 코드를 읽을 때, 내 머릿속의 상당 부분은 라이프타임(`'a`), 참조, 가변성을 추적하는 데 쓰인다.
실제로 이 부분에서 버그가 났을 때, 나는 이 코드가 **도메인 로직이 틀린 것인지, 아니면 단순히 대여 검사기를 통과하려고 억지로 우회한 것인지** 단번에 파악하기 힘들었다.

**반면 Nim은 어떨까:**
```nim
func explore(step: int; trace: BmcTrace;
             verdict: BmcVerdict;
             rule: TransitionRule): BmcVerdict {.raises: [].} =
```
여기서 코드를 읽는 나의 어텐션은 오직 '도메인 변수'와 '의미 제약'에만 쏠린다.
메모리를 어떻게 관리할 것인가에 대한 고민이 구문에서 떨어져 나가면서, 문법이 아니라 **인과관계 자체에 집중**할 수 있게 된다.

## '왜'를 강제하는 구조적 장치들

현업에서 가장 고통스러운 순간은 이전 작업자(혹은 어제의 나, 또는 LLM)가 "왜 이 코드를 이렇게 짰는지" 주석이나 커밋 메시지를 뒤져야 할 때다.

Nim은 이 '왜(Why)'를 언어 스펙 자체에 박아넣을 수 있게 해준다.

```nim
type
  SessionId = distinct string
  OwnerId = distinct string
```
`distinct`는 단순한 타입 별칭이 아니다. AI가 `SessionId`가 들어가야 할 자리에 `OwnerId`를 넣는 실수를 했을 때, 런타임 테스트까지 갈 것도 없이 컴파일러가 즉시 멱살을 잡는다.

또한, 함수가 어떤 전제 조건을 가지고 결과를 내는지 명시하는 계약(Contract)도 마찬가지다.
```nim
func toSessionId*(raw: string): Result[SessionId, AxiomError] {.
  raises: [],
  requires: raw.len > 0,
  ensures: (not result.isOk()) or
           (sessionIdString(result.get()) == raw)
.}
```
이건 단순한 문서화가 아니다. 컴파일러가 런타임 이전에 강제하는 계약이다.
물론 Rust에서도 매크로나 외부 크레이트를 쓰면 비슷하게 흉내 낼 수 있지만, Nim은 언어의 기본 철학 자체가 이런 **'의도의 가시화'**를 밀어붙인다. `func` 키워드 하나만으로 부작용(Side Effect)이 없음을 모듈 단위로 강제할 수 있는 것처럼 말이다.

## 결론: AI 시대의 진짜 병목

다시 말하지만, 성능이나 커뮤니티의 크기가 중요하다면 Rust는 의심의 여지 없이 훌륭한 선택이다. 메모리 안전성이라는 든든한 바닥을 제공하니까.

하지만 내가 Axiom을 만들며 뼈저리게 느낀 건, AI 보조 개발 시대에 진짜 무서운 리스크는 '안전하지 않은 코드'가 아니라 **'무엇을 의도했는지 실행 전에는 알 수 없는 코드'**라는 점이다.

코드를 읽지 않고도 작성자의 의도를 구조적으로 파악할 수 있는가?
이 질문에 대해 나는 타협하고 싶지 않았고, 그것이 때로는 투박해 보일지라도 내가 Nim을 선택한 이유다. 정합성이라는 천장을 뚫기 위해서.
