## 핵심 질문

AI가 짠 코드를 검증하는 엔진을 만들 때, 아키텍트는 골치 아픈 딜레마에 빠진다. 규칙은 어디에 둬야 하며, 누가 그걸 강제할 것인가?

Axiom의 결론은 단순하다. **규칙은 코드보다 먼저 정의되어야 한다.**

---

## Rule-First 원칙

보통의 소프트웨어 개발은 이런 식이다.

1. 일단 코드를 짠다.
2. 테스트 돌려서 버그를 잡는다.
3. 비슷한 버그가 안 나오게 규칙을 추가한다.
4. 무한 반복.

이런 반응형 방식은 AI가 만든 코드를 검증할 때 처참하게 깨진다. LLM은 테스트 케이스만 기가 막히게 통과하면서 정작 핵심 논리는 슬쩍 비껴가는 '그럴듯한 오답'을 만드는 데 천재다. 확률적 출력이라 버그가 미묘하게 다른 모습으로 재등장해서 테스트 우회도 쉽다.

규칙을 미리 박아두지 않으면, 어느 순간 엔진이 AI의 환각(Hallucination)에 가스라이팅 당해서 검증 표준을 스스로 낮춰버리는 대참사가 일어난다. 규칙 없이 그냥 코드 검사만 하다 보면, 결국 "AI가 만든 코드에서 규칙을 찾아내서" 검증하는 순환 논리에 빠진다. 자기 자신을 정당화하는 꼴이다.

코드를 보고 규칙을 찾아내는 검증 엔진은 결국 자기 정당화 루프에 빠질 뿐이다. 그래서 Axiom은 패러다임을 완전히 뒤집는다.

1. **규칙 선언**: 불변량, 계약, 안전 속성을 먼저 정의한다.
2. **인코딩**: 이 규칙을 검증 산출물로 변환한다.
3. **증명**: 코드가 규칙을 만족하는지 수학적으로 증명한다.
4. **실행 허용**: 증명된 후에만 실행을 허용한다.

## 헌법-조례-실록: 규칙의 계층화

Axiom은 규칙을 세 레이어로 나눈다. 법률 시스템에서 빌려온 구조다.

| 레이어 | 키워드 | 파일 패턴 | 목적 | 변경 빈도 |
|--------|--------|-----------|------|-----------|
| **헌법** | **Context** | `CONTEXT.md` | 절대 변하지 않는 설계 원칙 | 거의 없음 |
| **조례** | **Spec** | `docs/spec/*.md` | 상세 알고리즘 및 제약 조건 | 보통 |
| **실록** | **History** | `docs/history/*.md` | 결정 기록 및 히스토리 | 빈번 |

### 레이어 1: 헌법 (Context)

프로젝트의 정체성이다. "핵심 계산(Core)은 무조건 순수 함수(func)여야 한다" 같은 깨지면 안 되는 원칙들을 적는다. 여길 수정한다는 건 프로젝트의 뿌리를 흔드는 거라 아주 신중해야 한다.

```markdown
## 2. Core Architecture Principles (SSOT)

### Purity Hierarchy

- **Core Calculation Tier**: `src/core` 계산은 오직 `func`를 사용
- **Boundary Executor Tier**: `proc`는 OS/인터페이스에서만 존재
- **Executor-only Rule**: 경계 `proc`는 비즈니스 로직을 소유할 수 없다
```

### 레이어 2: 조례 (Spec)

실제 돌아가는 규칙들이다. SMT Solver 인터페이스나 Lean 4 증명 전략 같은 구체적인 명세가 여기 담긴다. 구현할 때 가이드라인이 된다.

- `bmc_core.md`: 모델 체킹 알고리즘, Z3 인터페이스, 타임아웃 규칙
- `uap_logic.md`: 범용 파이프라인 단계, 반례 승격, 불변량 정제
- `lean_proof.md`: Lean 4 의미론, tactic 전략, 재생 프로토콜

### 레이어 3: 실록 (History)

"왜 그때 그렇게 짰지?"에 대한 답이다. 과거의 타협이나 결정 맥락을 기록한다. 실록은 참고용일 뿐, 현재의 구현을 지배하는 규칙이 되어서는 안 된다.

```markdown
## 2026-03-26
- Core는 Bottom-up Sorter, Vault Manager, Axiom Orchestration Loop를 도입했다
- Stale axiom 차단과 Vault 기반 승격 경로를 확립했다
```

실록은 규칙과 분리된다. "왜 이 규칙이 생겼나?" 궁금하면 실록을 찾아본다. 하지만 코드를 짤 땐 헌법과 조례만 보면 된다. 역사적 잡음에 안 빠지게.

## 무결성 하향 조정 금지: 정합성 강선

Axiom에서 가장 엄격하게 관리하는 원칙은 하나다. **무결성은 증가만 가능하고 감소는 불가능하다.**

### func → proc 트랩

Nim 언어는 순수 함수(`func`)와 부작용이 있는 절차(`proc`)를 엄격히 나눈다. 개발하다 보면 유혹이 생긴다.

> "아, 여기 `echo` 하나만 찍어보게 잠시만 `proc`로 바꿀까?"

**Axiom에서는 안 통한다.**

왜냐하면:

1. **하향 조정은 일방통행**: `proc`가 생기면 개발자는 계속 부작용을 추가한다. 잡초처럼 퍼진다.
2. **순수성은 이진법**: '약간 불순한 함수' 같은 건 없다. 순수하거나 아니거나다.
3. **증명은 순수성 필수**: 부작용이 예측 불가능한 함수에 수학적으로 속성을 증명할 수 없다.

Nim 2.0+의 `strictFuncs` 모드와 effect 추적이 이 구분을 더 강력하게 만든다:

```nim
# strictFuncs는 컴파일 타임에 순수성을 강제한다
{.push strictFuncs.}
func verifyLogic(node: Node): Result[void, Error] {.raises: [], tags: [].} =
  # 컴파일러가 외부 상태 변경이나 IO 호출을 거부한다
  if node.isValid: ok() else: err(InvalidNode)
{.pop.}
```

이게 Nim이 Axiom 아키텍처에 최적화된 이유다. 언어 자체가 컴파일 타임에 순수성 경계를 강제한다. 코드 리뷰에서가 아니라.

### Axiom의 강제 메커니즘

Axiom은 **순수성 계층**을 유지한다:

```
┌─────────────────────────────────────────────────────────────┐
│ Core Calculation Tier (src/core)                            │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ func-only zone: 계산, 파싱, 정규화                      │ │
│ │ 반환: Result[T], Option[T], VerifiedType[T]            │ │
│ │ 금지: IO, 변경, 예외                                     │ │
│ └─────────────────────────────────────────────────────────┘ │
│                                                             │
│ Boundary Executor Tier (src/ui, src/cli, gateway)          │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ proc-allowed zone: OS, TTY, 파일시스템, 프로세스         │ │
│ │ 역할: 핵심 func 결과 실행, 의존성 주입                   │ │
│ │ 금지: 비즈니스 로직, 도메인 규칙                         │ │
│ └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

`src/core`에 `proc`가 나타나면 **설계 부채**로 표시된다. 필수 리팩터링 계획이 있는 일시적 타협. 코드는 다음이 완료될 때까지 배포되지 않는다:

1. 실행자 경계가 명확히 표시된다
2. 백로그에 "func 승격" 작업이 존재한다
3. 불순성의 이유가 `CONTEXT.md`에 문서화된다

### 이 규칙을 위반하면 어떻게 되는가

Axiom에서 `func`를 `proc`로 하향 조정하는 건 단순히 코드를 수정하는 행위가 아니다. **시스템의 수학적 무결성을 파괴하는 행위**로 간주된다. 정당한 사유(`CONTEXT.md` 업데이트) 없는 하향 조정은 CI 단계에서 원천 차단된다.

Axiom의 CI는 **Procedure Gate**를 통해 이를 강제한다:

1. `src/core` 파일이 변경되면 `proc` 추가 확인
2. diff에 `func`→`proc` 변환이 나타나면 **커밋 실패**
3. 해당 문서 없이 `proc`가 존재하면 **커밋 실패**

유일한 예외: `--style-only` 변경(오타, 주석, 공백). 그 외 모든 것은 아키텍처적 정당화가 필요하다.

컴파일 타임에 Nim의 effect 시스템은 추가 강제를 제공한다:

```nim
# [Layer 1: Core Calculation Tier]
{.push strictFuncs.}
func verifyNodeIntegrity(node: AxiomNode): Result[void, IntegrityError] {.raises: [].} =
  ## IO 없음, 변경 없음, 예외 없음 - 수학적으로 순수
  if node.fingerprint == node.computedFingerprint():
    ok()
  else:
    err(IntegrityError.FingerprintMismatch)

# [Layer 2: Boundary Executor Tier]
proc executeVerification(node: AxiomNode, output: File) {.raises: [IOError].} =
  ## 얇은 실행자: 의존성을 주입하고 모든 로직은 func에 위임
  let result = verifyNodeIntegrity(node)  # 순수 함수 호출
  if result.isOk:
    output.write("검증 통과\n")  # IO는 경계에 격리
  else:
    output.write("검증 실패: " & $result.error & "\n")
{.pop.}
```

패턴이 보이는가? `verifyNodeIntegrity`는 순수 수학이다. `executeVerification`은 IO를 처리하는 얇은 래퍼다. 비즈니스 로직은 절대 불순 영역으로 넘어가지 않는다.

### 실제 예시

리포트 렌더링을 구현할 때 Axiom은 선택에 직면했다:

```nim
# 처음에 유혹받는 접근:
proc renderReport(data: seq[ProofResult]) =
  # 직접 IO: stdout에 기록
  for r in data:
    echo r.format()

# 강제된 순수성 접근:
func renderReportLines(data: seq[ProofResult]): seq[string] {.noSideEffect.} =
  # 순수 계산: 문자열 반환
  for r in data:
    result.add r.format()

proc writeReport(data: seq[ProofResults]) =
  # 얇은 실행자: 출력 스트림 주입
  let lines = renderReportLines(data)
  for line in lines:
    echo line
```

두 번째 접근은 관심사를 분리한다:

- `renderReportLines`는 순수하고, 테스트 가능하며, 검증 가능하다
- `writeReport`는 파일 출력, 네트워크 출력, 또는 테스트 목으로 교체할 수 있는 간단한 실행자다

이건 과도한 엔지니어링이 아니다. **검증을 가능하게 만드는 것이다.**

## 이 원칙들이 존재하는 이유

Axiom은 AI가 생성한 코드를 검증한다. 이걸 신뢰성 있게 하려면 검증하는 코드보다 **더 엄격해야 한다**.

Rule-First 설계가 보장하는 건:

- 검증 규칙이 기존 코드를 정당화하려고 후속 조정되지 않는다
- 아키텍처 원칙이 명시적이고, 발견 가능하며, 강제된다
- 무결성은 한 방향으로만 이동한다: 상향으로

헌법-조례-실록 분리가 보장하는 건:

- 원칙이 세부 사항에 잠기지 않고 가시적으로 유지된다
- 구현자가 명세를 쉽게 찾는다
- 역사가 정보를 제공하지만 혼잡하게 하지 않는다

무결성 하향 조정 금지 정책이 보장하는 건:

- 순수성이 편의를 위해 희생되지 않는다
- 검증이 수학적으로 건전하게 유지된다
- 기술 부채가 가시적이고, 추적되며, 일시적이다

---

## 결론

이 원칙들은 열망적이지 않다. CI 게이트, 코드 리뷰, 아키텍처 문서가 강제한다. Axiom의 모든 커밋은 Procedure Gate를 통과해야 한다. 핵심의 모든 `proc`는 존재를 정당화해야 한다. 모든 규칙 변경은 구현 전에 문서화되어야 한다.

> **아키텍처 자체가 검증 가능해야 한다.**

결국 검증 엔진을 만든다는 건, **확률적 코드를 확률적 도구로 검증할 수 없다**는 뜻이다. 기반은 결정론적이고, 수학적으로 증명되며, 아키텍처적으로 순수해야 한다.

다음 포스트에서는 이 수학적 약속을 어떻게 실제 증명으로 변환하는지, 경계 모델 체킹(BMC) 구현을 다뤄보겠다.
