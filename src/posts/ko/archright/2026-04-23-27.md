AI가 코드를 작성하는 시대다.

문제는 더 이상 생산성이 아니다.

신뢰다.

코드는 빠르게 생성된다
테스트도 통과한다
하지만

이 코드가 정말 맞는지 보장할 수 없다

<hr />

## 1) 우리가 만들고 있는 것: Axiom

Axiom은 단순한 "자동 증명 엔진"이 아니다.

AI가 만든 코드를 '믿을 수 있게' 만드는 시스템이다.

이를 위해 우리는:

- **SMT Solver (Z3)**
- **추상화 (CEGAR)**
- **요약 기반 분해 (Summary)**
- **형식 검증 (Lean 4)**

을 하나의 orchestration으로 묶었다.

<hr />

## 2) 사건의 발단: "42%의 침묵"

`ax prove`를 실행하면, 진행률이 정확히 42%에서 멈추는 현상이 발생했다.

이건 단순한 UI glitch처럼 보였다.

하지만 실제로는 세 가지 가능성이 동시에 존재했다:

- 실제 deadlock
- solver가 끝없이 계산 중
- 일부는 끝났지만 UI가 보여주지 못함

즉,

**시스템이 죽었는지, 생각 중인지 구분할 수 없는 상태**

<hr />

## 3) Kimi의 진단: "이건 진짜 멈춘 거다"

조사 결과, 실제 버그가 있었다.

**원인 1 — 순차적 결과 대기**

```nim
let result = ^task  # spawn 순서대로 block
```

→ 하나가 hang되면 전체가 멈춤

**원인 2 — blocking I/O**

```nim
readLine()  # Z3 pipe에서 무한 대기 가능
```

→ solver 상태와 무관하게 deadlock 가능

<hr />

## 4) 1차 해결: 시스템을 멈추지 않게 만든다

우리는 두 가지를 수정했다.

✔ **Out-of-order 처리**

```nim
if isReady(task):
  collect(task)
```

→ 완료된 것부터 즉시 반영

✔ **Non-blocking I/O**

```nim
readData()  # chunk 기반 처리
```

→ solver 종료/이상 상태 즉시 감지

이걸로 "진짜 freeze"는 제거됐다.

<hr />

## 5) 그런데 더 큰 문제가 남았다

버그를 고친 뒤에도 동일한 질문이 남았다.

**"solver가 영원히 답을 못 찾는 경우는 어떻게 할 것인가?"**

<hr />

## 6) 기술 논쟁: "우회인가, 포위인가?"

여기서 두 가지 관점이 충돌했다.

🏛️ **제미나이 — 공학적 포위 전략**

> "우리는 답을 찾는 게 아니라 증거를 모은다"

- CEGAR로 문제를 단순화하고
- Summary로 쪼개고
- Lean 4로 최종 확정한다

🏛️ **Kimi — 한계 인식 기반 설계**

> "이 도구들은 모두 실패한다"

- SMT는 undecidable
- CEGAR도 반복 중 멈출 수 있음
- Lean 자동화도 결국 search 문제

**결론:**

**완벽한 해결책은 없다**

<hr />

## 7) 우리가 내린 결론

이 논쟁에서 얻은 핵심은 이것이다:

❌ "항상 증명하는 시스템"은 불가능하다
✅ "실패를 통제하는 시스템"은 가능하다

<hr />

## 8) Axiom의 설계 원칙

Axiom은 "증명 엔진"이 아니라:

**"실패를 구조화하는 시스템"**이다

🏛️ **L1/L2 — 실패를 멈춘다**

- timeout
- abstraction (CEGAR)

→ 무한 계산 차단

🏛️ **L3 — 실패를 쪼갠다**

- decomposition
- summary

→ 전체 실패 → 부분 성공

🏛️ **L4 — 실패를 넘긴다**

- Lean 4
- 인간 invariant

→ 자동화 → 협업 전환

<hr />

## 9) 그리고 드러난 또 다른 문제

버그는 사라졌지만, 새로운 문제가 보였다.

**여전히 "멈춘 것처럼 보인다"**

이건 단순 UX 문제가 아니다.

**핵심 문제**

검증 시스템에서 '보이지 않는 진행'은 곧 신뢰 붕괴다

<hr />

## 10) 해결: Heartbeat 패치

우리는 접근을 바꿨다.

> "정확한 진행률"보다
> "살아있다는 증거"를 보여준다

**Before**

```
[████████░░] 42%
```

**After**

```
[████████░░] 42%

[VERIFIED] expr.nim::add
[VERIFIED] results.nim::isErr
[TIMEOUT] main.nim::getStr
```

이 변화의 의미:

**"멈춘 게 아니라, 어려운 문제를 풀고 있다"**

<hr />

## 11) 추가 개선: Evidence 기반 힌트

Z3는 timeout 시 아무 말도 하지 않는다.

그래서 우리는 우회했다:

- **구조적 복잡도 분석 (Topology)**
- **Lean 미해결 goal 추출**

결과:

"왜 실패했는지"를 직접 말하진 못해도
"어디가 문제인지"는 보여줄 수 있다

<hr />

## 12) Lessons Learned

이번 사건은 두 가지를 명확히 했다.

✔ **Nim threadpool의 한계**

- FlowVar cancellation 불가
- polling 기반 설계 필수

✔ **Z3의 블랙박스 특성**

- timeout = unknown
- 원인 추적 불가

→ 정황 기반 진단 필요

<hr />

## 13) 결론

이번 "42% 사건"은 단순한 버그가 아니었다.

**AI 코드 검증 시스템이 반드시 마주하는 구조적 한계의 표면화였다**

우리는 여기서 한 가지를 선택했다.

**완벽한 증명을 포기하고**
**통제 가능한 실패를 선택했다**

한 문장 정의:

> 검증 엔진에서 중요한 것은 정답이 아니라,
> 실패가 어떻게 드러나고 통제되는가다.

<hr />

## 마무리

Axiom은 이제 42%에서 멈추지 않는다.

그리고 더 중요한 변화는 이것이다:

**멈추더라도**
**왜 멈췄는지, 다음에 무엇을 해야 하는지 말해준다**

한 줄 요약:

> Axiom은 코드를 증명하는 엔진이 아니라,
> 신뢰를 생성하는 시스템이다.
