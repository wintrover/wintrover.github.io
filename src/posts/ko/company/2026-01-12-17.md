AI 에이전트는 제 개발 속도를 하룻밤 사이에 바꿔놓았습니다. 예전엔 일주일 걸리던 코드 양을 하루에 밀어 넣을 수 있게 됐고, 겉보기엔 좋은 일처럼 들립니다. 하지만 작은 엣지 케이스 하나가 전체 플로우를 쓰러뜨리는 순간, 그 속도는 곧바로 부메랑이 됩니다.

그 속도에서 QA는 경쟁력이 되거나, 상시 소방서가 됩니다. 저는 전자를 선택했고, `d:\Coding\Company\Ochestrator`에서 테스트 접근을 “테스트를 더 쓰기”가 아니라 **테스트 설계(test design)** 관점으로 재구성했습니다.

- TDD
- EP-BVA(동치 분할 + 경계값 분석)
- Pairwise(조합 테스트)
- 상태 전이(State Transition) 테스트

## 1. “테스트를 더 쓰기”가 아니라 “테스트 설계”가 필요했던 이유

코드량이 늘면 문제는 단순히 “커버리지”가 아닙니다. 실제 문제는 가능한 입력과 상태의 공간이 제 시간을 훨씬 빠르게 앞지른다는 점입니다.

그래서 저는 이런 질문을 멈췄습니다.

- “이 함수 테스트를 썼나?”

그리고 이런 질문을 하기 시작했습니다.

- “실패 표면을 대표하는 테스트 케이스를 선택했나?”

이 관점이 구조적인 테스트 설계 기법들로 저를 이끌었습니다.

---

## 2. TDD: 첫날부터 테스트 가능하게 설계하기

**원리**: TDD(Test-Driven Development)는 “코드를 먼저 쓰고 테스트를 나중에”라는 흐름을 뒤집습니다. **Red-Green-Refactor** 사이클을 따릅니다.
1. **Red**: 새로운 요구사항에 대한 테스트를 먼저 만들고 실패하는지 확인한다. 테스트가 실제로 뭔가를 검증하고 있고, 요구사항이 이미 충족된 상태가 아니라는 것을 보장한다.
2. **Green**: 테스트를 통과시키기 위한 최소한의 코드를 작성한다. 이 단계에서 과도한 설계는 피한다.
3. **Refactor**: 테스트가 초록 상태를 유지하는 범위에서 코드를 정리한다.

**Orchestrator에서**
AI 에이전트가 복잡한 비즈니스 로직을 빠르게 생성할 수 있는 만큼, 저는 TDD를 통해 로직이 “처음부터 테스트 가능”하도록 강제했습니다. 예를 들어 Temporal 워크플로의 `RetryPolicy`를 구현할 때, 정책 로직 한 줄 쓰기 전에 지수 백오프(exponential backoff)의 테스트 케이스부터 작성했습니다.

```python
# Simplified TDD Example for Retry Logic
def test_retry_interval_calculation():
    policy = ExponentialRetry(base_delay=1.0, max_delay=10.0)
    # 1st attempt: 1.0s
    assert policy.get_delay(attempt=1) == 1.0
    # 2nd attempt: 2.0s
    assert policy.get_delay(attempt=2) == 2.0
    # Capped at 10.0s
    assert policy.get_delay(attempt=10) == 10.0
```

이 방식은 “지연 시간 계산”과 “재시도 실행”을 분리하게 만들었고, 결과적으로 모듈성이 높고 견고한 시스템으로 이어졌습니다.

## 3. EP-BVA: 수학적으로 효율적인 케이스 선택

**원리**
- **동치 분할(EP)**: 모든 값을 테스트하는 대신, 시스템이 동일하게 동작할 것으로 기대되는 입력 도메인을 여러 그룹(분할)으로 나누고 각 그룹에서 대표값만 테스트한다.
- **경계값 분석(BVA)**: 버그는 대개 분할의 “경계”에 숨어 있다. 경계값과 경계 근처(안/밖)를 집중적으로 테스트한다.

**Orchestrator에서**
사용자 업로드 파일 크기에 엄격한 제한이 있었습니다(예: 1MB ~ 10MB).
- **분할**
  - Invalid(< 1MB)
  - Valid(1MB - 10MB)
  - Invalid(> 10MB)
- **경계 포인트**: 0.99MB, 1.0MB, 1.01MB, 9.99MB, 10.0MB, 10.01MB

현업에서 정말 효과적이었던 사례는 **bcrypt의 72바이트 제한**이었습니다. 많은 개발자가 bcrypt가 72바이트 이후 문자를 무시한다는 사실을 모르고, 길이 정책을 잘못 설계합니다.

```python
# apps/backend/tests/test_auth_service.py
def test_password_length_boundaries(self, auth_service):
    # Boundary: 72 bytes
    p72 = "a" * 72
    h72 = auth_service.get_password_hash(p72)

    # Just above the boundary: 73 bytes
    p73 = p72 + "b"
    # Bcrypt will treat p73 the same as p72 if only the first 72 bytes are used
    assert auth_service.verify_password(p73, h72) is True
```

이런 포인트에 집중하면, 수백 개의 가능성을 단 6~10개의 “효율 좋은” 케이스로 줄일 수 있습니다.

## 4. Pairwise: 조합 폭발을 길들이기

**원리**: 대부분의 버그는 단일 파라미터 또는 **두 개 파라미터의 상호작용**에서 발생합니다. Pairwise 테스트는 모든 파라미터 쌍의 조합을 최소 1회 이상 포함하도록 테스트 케이스를 구성해, 케이스 수를 크게 줄이면서 결함 탐지력을 유지합니다.

**Orchestrator에서**
AI 추론 엔진은 여러 설정 축이 있습니다.
- **Execution Provider**: `[CUDA, CPU, OpenVINO]` (3)
- **Model Size**: `[Small, Medium, Large]` (3)
- **Quantization**: `[INT8, FP16, FP32]` (3)
- **Async Mode**: `[Enabled, Disabled]` (2)

전체 조합은 $3 \times 3 \times 3 \times 2 = 54$ 케이스입니다. Pairwise로는 대략 **12~15 케이스**로 대부분의 결함을 잡을 수 있습니다.

```python
# Using allpairspy to generate the matrix
from allpairspy import AllPairs

parameters = [
    ["CUDA", "CPU", "OpenVINO"],
    ["Small", "Medium", "Large"],
    ["INT8", "FP16", "FP32"],
    ["Enabled", "Disabled"]
]

for i, combo in enumerate(AllPairs(parameters)):
    print(f"Test Case {i}: {combo}")
```

이렇게 하면, PR마다 54개 조합 전체를 돌리지 않고도 하드웨어 호환성 매트릭스에 대한 높은 신뢰를 유지할 수 있습니다.

## 5. 상태 전이 테스트: 프로세스의 생애를 모델링하기

**원리**: 시스템의 동작이 현재 상태와 이벤트에 따라 달라지는 경우, 상태 전이 다이어그램을 만들고 다음을 검증합니다.
1. 모든 유효 전이가 가능해야 한다.
2. 모든 무효 전이는 제대로 차단되어야 한다(네거티브 테스트).
3. 시스템이 올바른 최종 상태에 도달해야 한다.

**Orchestrator에서**
KYC 검증 워크플로는 복잡한 상태 머신입니다. 문서의 상태는 아래 흐름을 따릅니다.
`PENDING` $\rightarrow$ `UPLOADING` $\rightarrow$ `PROCESSING` $\rightarrow$ `VERIFIED` 또는 `REJECTED`

저는 `REJECTED` 상태가 `PROCESSING`을 거치지 않고 곧장 `VERIFIED`로 점프할 수 없도록 하는 테스트를 만들었습니다.

```python
# apps/backend/tests/test_integration_kyc_workflow.py
def test_invalid_state_transitions(workflow_engine):
    workflow_engine.set_state(ImageStatus.REJECTED)

    # This should be blocked by the business logic
    with pytest.raises(IllegalStateError):
        workflow_engine.transition_to(ImageStatus.VERIFIED)
```

AI 에이전트가 로직을 “단축(short-circuit)”하려고 시도하는 경우도 있기 때문에, 상태 머신을 엄격히 검증하는 테스트는 전체 비즈니스 프로세스 무결성에 매우 중요합니다.

---

## Conclusion

AI 에이전트 시대에 코드는 싸졌습니다. 신뢰는 그렇지 않습니다.

QA가 무너지지 않도록 지켜준 것은 “더 많은 테스트”가 아니라, 코드량이 늘어도 확장 가능한 테스트 설계 기법들이었습니다.

- TDD: 빠른 피드백과 안전한 리팩터링
- EP-BVA: 엣지 케이스를 체계화
- Pairwise: 조합 폭발을 제어
- 상태 전이 테스트: 실제 워크플로 검증

이 테스트 도구 상자는 코드 생산량이 계속 늘어도 앞으로도 계속 쓸 것 같습니다.
