분산 시스템에서 비동기 태스크를 처리할 때 Celery + Redis 조합은 가장 흔한 선택지입니다. 저 역시 KYC(Know Your Customer) 오케스트레이터의 초기 설계에서 익숙함 때문에 Celery를 선택했습니다. 하지만 서비스 복잡도가 커질수록 거대한 벽에 부딪혔습니다. 바로 **멱등성(idempotency)** 보장과 **복잡한 상태(state)** 관리였습니다.

이 글에서는 왜 Celery에서 Temporal로 옮기게 되었는지, 그리고 그 과정에서 멱등성을 어떻게 확보했는지를 기술적으로 정리해보려 합니다.

---

## 1. Celery/Redis의 한계: 왜 바꿔야 했나?

### 멱등성 관리의 어려움

Celery는 “Fire and Forget” 스타일의 태스크에 매우 적합하지만, 네트워크 오류나 워커 장애로 재시도가 발생할 때 **중복 실행** 위험이 큽니다. 특히 GPU를 많이 소모하는 얼굴 인식 태스크에서는 중복 실행이 비용/성능 측면에서 치명적이었습니다.

### 상태의 파편화

KYC 프로세스는 아래 순서를 따릅니다.
1. 사용자가 신분증 이미지를 업로드한다.
2. 사용자가 셀피 비디오를 업로드한다.
3. 두 파일이 모두 준비되면 얼굴 유사도를 비교한다.

![Celery Sequence Diagram](/images/2026-01-08-16-celery-sequence.png)

Celery 환경에서는 이미지와 비디오 업로드가 언제 끝날지 모르기 때문에, 매번 DB를 조회하거나 중간 상태를 Redis에 저장하는 복잡한 로직이 필요했습니다. “파일이 모두 모였는가?”를 확인하는 로직이 여러 곳에 흩어져 유지보수가 어려워졌습니다.

---

## 2. Temporal 도입: 오케스트레이션의 패러다임 전환

Temporal은 단순한 메시지 큐가 아니라 **Stateful Workflows** 엔진입니다.

### 워크플로 로직은 “결정적(deterministic)”이어야 한다

Temporal 워크플로 코드는 “리플레이(replay)”를 전제로 동작하기 때문에, 동일한 입력/히스토리에 대해 항상 같은 워크플로 API 호출 시퀀스를 만들어야 합니다. 따라서 워크플로 내부에서 네트워크 I/O, 파일 I/O, 시스템 시간(`DateTime.now` 같은), 랜덤, 스레딩 같은 “외부 세계 의존” 부작용을 직접 수행하면 안 됩니다. 이런 작업은 액티비티(activity)로 밀어내고, 워크플로는 오케스트레이션 자체에만 집중해야 합니다.

공식 문서: https://docs.temporal.io/develop/python/core-application#workflow-logic-requirements

### 워크플로 중심 설계

Temporal을 도입한 뒤 가장 크게 달라진 점은 비즈니스 로직의 가시성입니다. `FaceSimilarityWorkflow`는 파일이 준비될 때까지 우아하게 기다릴 수 있습니다.

![Temporal Workflow Diagram](/images/2026-01-08-16-temporal-workflow.png)

```python
# Core logic of FaceSimilarityWorkflow
@workflow.run
async def run(self, data: SimilarityData) -> SimilarityResult:
    # Wait up to 1 hour until both image and video are collected
    await workflow.wait_condition(
        lambda: any(f["type"] == "image" for f in self._files)
        and any(f["type"] == "video" for f in self._files),
        timeout=timedelta(hours=1),
    )

    # Execute GPU activity once all files are ready
    result = await workflow.execute_activity(
        check_face_similarity_activity,
        activity_data,
        retry_policy=RetryPolicy(maximum_attempts=3)
    )
    return result
```

이 코드는 `workflow.wait_condition`을 사용해 조건이 만족될 때까지 워크플로를 중단(suspend)하지만, 이벤트 루프를 블로킹하지 않습니다. Celery로는 이런 동작을 만들기 위해 복잡한 폴링이나 웹훅 로직이 필요했습니다.

## 3. 멱등성 전략: 이중 방어선 만들기

Temporal을 쓰더라도, 액티비티 레벨에서의 멱등성은 여전히 중요합니다. 저는 아래와 같은 “이중 방어” 전략을 만들었습니다.

### Step 1: Temporal의 기본 보장

Temporal은 워크플로 진행 상황을 이벤트 히스토리로 기록합니다. 따라서 워커가 재시작하더라도 마지막 성공 지점부터 정확히 이어서 진행합니다.

### Step 2: 액티비티 내부의 명시적 체크

Temporal 액티비티는 “at-least-once” 실행 모델을 따르기 때문에, 워커가 액티비티를 성공적으로 수행한 직후 서버에 알리기 전에 죽으면 액티비티가 재시도될 수 있습니다. 그래서 공식 문서에서도 액티비티를 멱등적으로 만들 것을 강하게 권장합니다.

![Activity Idempotency Diagram](/images/2026-01-08-16-activity-idempotency.png)

공식 문서: https://docs.temporal.io/develop/python/error-handling#make-activities-idempotent

실무에서는 아래 두 가지를 함께 사용했습니다.
- 외부 시스템 호출에는 워크플로 실행/액티비티 식별자를 조합한 **idempotency key**를 전달
- 내부적으로는 DB에서 unique key로 중복 저장/중복 처리를 방지(또는 기존 결과 존재 여부 확인)

```python
@activity.defn
async def check_face_similarity_activity(data: SimilarityData) -> SimilarityResult:
    info = activity.info()
    idempotency_key = f"{info.workflow_run_id}-{info.activity_id}"
    session_id = data["session_id"]

    with get_db_context() as db:
        existing = (
            db.query(FaceSimilarity)
            .filter(FaceSimilarity.idempotency_key == idempotency_key)
            .first()
        )
        if existing:
            return SimilarityResult(success=True, message="Already processed.")

```

## 4. 결과: 무엇이 달라졌나?

| 비교 항목 | Celery/Redis 기반 | Temporal 기반 |
| :--- | :--- | :--- |
| **상태 관리** | DB/Redis에 수동 저장 | 엔진이 자동으로 관리 |
| **재시도 전략** | 수동 exponential backoff | 선언적 Retry Policy |
| **가시성** | 로그를 파고들어야 함 | Temporal UI에서 히스토리 확인 |
| **멱등성** | 보장하기 매우 어려움 | 구조적으로 달성 가능 |

## Conclusion

Celery에서 Temporal로의 전환은 단순히 도구를 바꾼 게 아니라, **비즈니스 프로세스를 코드로 정의하는 방식 자체**를 바꾼 일이었습니다. 특히 멱등성이 최우선인 금융/인증 시스템에서는 Temporal이 제공하는 안정성이 대체 불가능하다고 느꼈습니다.

복잡한 비동기 로직과 멱등성 문제 때문에 밤잠을 설치고 있다면, Temporal로의 마이그레이션을 강력히 추천합니다.
