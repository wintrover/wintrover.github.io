## Introduction

처음에는 단순히 “개발 세팅 좀 최적화해볼까?”라는 마음이었습니다. 하지만 결과적으로는 몇 달 동안의 생산성을 좌우한 **인프라 전면 개편**으로 이어졌습니다. 포트 충돌, 느린 빌드, 뒤엉킨 의존성 때문에 개발 환경 자체가 병목이었고, 저는 결국 Traefik 기반 중앙 오케스트레이션과 Turborepo 모노레포 구조로 마이그레이션했습니다. 이 글은 그 과정에서 겪었던 시행착오와, 뼈저리게 배운 교훈들을 정리한 기록입니다.

## 1. 임계점: 개발이 악몽이 되던 순간

### 매일 반복되던 고통

매일 아침, 같은 의식이 반복됐습니다.

| 문제 | 증상 | 하루 낭비 시간 |
|---------|---------|-------------------|
| 포트 충돌 | "백엔드가 안 떠요. 8000 포트 누가 쓰고 있어요" | 15-30분 |
| 빌드 병목 | "프론트 빌드 기다리는 중... 또" | 20-45분 |
| 의존성 지옥 | "Module not found: kyc_core.utils" | 10-20분 |
| 환경 드리프트 | "내 컴에서는 되는데요" | 30-60분 |
| | | |

인프라 문제로만 하루 1.5~2시간을 잃고 있었습니다. 그때 “이건 더는 안 된다”는 결론을 내렸습니다.

### 첫 시도는 왜 실패했나

처음 접근은 너무 순진했습니다. “포트 충돌만 해결하면 되겠지.”

```bash
# The "quick fix" that made everything worse
docker-compose up -d --scale backend=2 --scale frontend=2
```

결과는 완전한 혼돈이었습니다. 서비스 간 통신이 끊기고, DB는 서로 충돌했고, 개발 환경이 그대로 붕괴했습니다.

**교훈**: 전신 질환을 반창고로 치료할 수는 없습니다.

## 2. Traefik 여정: 포트 지옥에서 오케스트레이션 천국으로

### 조사 단계

역프록시/게이트웨이 솔루션을 비교했습니다.

| 솔루션 | 장점 | 단점 |
|----------|------|------|
| Nginx | 안정적, 문서 많음 | 설정이 복잡, 서비스 디스커버리를 수동으로 해야 함 |
| HAProxy | 고성능 | 러닝커브가 가파름, Docker 친화적이지 않음 |
| Traefik | Docker 네이티브, 자동 서비스 디스커버리 | 생태계가 비교적 새로움, 튜토리얼이 적음 |

Docker 네이티브 접근과 자동 서비스 디스커버리를 이유로 Traefik을 선택했습니다.

### 첫 Traefik 구현(실패)

```yaml
# My initial traefik.yml - full of mistakes
version: '3.8'

services:
  traefik:
    image: traefik:v2.11
    ports:
      - "80:80"  # ❌ BAD: Conflicted with existing services
      - "8080:8080"  # ❌ BAD: Hardcoded port
    # ❌ MISSING: Network configuration
    # ❌ MISSING: Docker provider configuration
```

**결과**: 완전 실패. 서비스가 서로 통신하지 못했고, 기존 셋업까지 깨뜨렸습니다.

### 돌파구: Docker 네트워크를 이해하기

며칠 동안 디버깅한 끝에 핵심 원인을 깨달았습니다. “나는 Docker 네트워크 방식으로 사고하지 않고 있었다.”

![Docker Network Architecture](/images/2025-11-30-14-docker-network.png)

### 동작하는 Traefik 구성

```yaml
# The final working version
version: '3.8'

services:
  traefik:
    image: traefik:v2.11
    container_name: traefik-proxy
    command:
      - --api.dashboard=true
      - --providers.docker=true
      - --providers.docker.exposedbydefault=false
      - --providers.docker.network=traefik_proxy
      - --providers.file.filename=/etc/traefik/dynamic.yml
      - --entrypoints.web.address=:80
      - --entrypoints.websecure.address=:443
      - --log.level=DEBUG
    ports:
      - 80:80
      - 443:443
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - ./certs:/etc/traefik/certs:ro
      - ./traefik_dynamic.yml:/etc/traefik/dynamic.yml:ro
    networks:
      - traefik_proxy
    labels:
      - traefik.enable=true
      - traefik.http.routers.traefik.rule=Host("traefik.127.0.0.1.sslip.io")
      - traefik.http.routers.traefik.entrypoints=web
      - traefik.http.routers.traefik.service=api@internal
    restart: unless-stopped

networks:
  traefik_proxy:
    external: true
    name: traefik_proxy
```

**핵심 돌파구**: 네트워크를 인라인으로 정의하는 대신, `external`로 만들어두고 각 서비스가 여기에 조인하게 한 것입니다.

### 병렬 개발의 혁명

Traefik을 깔아두니, 드디어 여러 환경을 동시에 띄울 수 있게 됐습니다.

```bash
# Environment 1: Feature branch
docker-compose -f docker-compose.feature.yml up -d

# Environment 2: Hotfix branch
docker-compose -f docker-compose.hotfix.yml up -d

# Environment 3: Testing branch
docker-compose -f docker-compose.testing.yml up -d
```

각 환경은 서로 다른 서브도메인을 쓰고, 포트 충돌이 없으며, 완전히 격리됩니다.

## 3. Turborepo 마이그레이션: 서브모듈 지옥에서 모노레포 천국으로

### 서브모듈 악몽

예전 구조는 Git 서브모듈 투성이였습니다.

```
project-root/
├── backend/ (git submodule)
├── frontend/ (git submodule)
├── shared-utils/ (git submodule)
└── kyc-core/ (git submodule)
```

**매일 터지던 문제**
- 서브모듈 sync 실패: `fatal: reference is not a tree`
- 의존성 버전 충돌
- 공유 빌드 파이프라인 부재
- 서비스 간 통합 테스트 불가

### Turborepo 조사

모노레포 도구도 비교했습니다.

| 도구 | 러닝커브 | 빌드 성능 | 생태계 |
|------|----------------|-------------------|-----------|
| Lerna | 높음 | 중간 | 성숙 |
| Nx | 매우 높음 | 매우 좋음 | 복잡 |
| Turborepo | 중간 | 매우 좋음 | 성장 중 |

단순성과 빌드 성능을 이유로 Turborepo를 선택했습니다.

### 마이그레이션에서 아팠던 지점들

#### 1) Docker 볼륨 마운트 이슈

```dockerfile
# ❌ BROKEN: Path didn't exist in container
VOLUME ["/app/packages/kyc_core"]

# ✅ FIXED: Correct path after restructuring
VOLUME ["/app/packages/kyc-core"]
```

#### 2) PYTHONPATH 설정 지옥

```bash
# ❌ BROKEN: Python couldn't find shared packages
export PYTHONPATH=/app/packages/kyc_core

# ✅ FIXED: Added all package paths
export PYTHONPATH=/app/packages/kyc-core:/app/packages/shared-utils
```

#### 3) Vite HMR이 안 됨

```typescript
// vite.config.ts - The HMR fix
export default defineConfig({
  server: {
    fs: {
      allow: ['../../packages'] // Allow Vite to access packages
    },
    watch: {
      usePolling: true // Required for Docker environments
    }
  }
});
```

## 4. 결과: 정말 가치가 있었나?

### Before vs After

| 지표 | Before | After | 개선 |
|--------|--------|-------|-------------|
| 환경 셋업 시간 | 45-60분 | 5-10분 | 85% 단축 |
| 전체 빌드 시간 | 8-12분 | 2-3분 | 75% 단축 |
| 인프라로 잃는 시간 | 1.5-2시간 | 10-15분 | 90% 감소 |
| 병렬 개발 환경 | 1개(충돌) | 3개+(격리) | 300% 향상 |

### 예상치 못한 이점

1. **크로스 팀 협업**: 서로 다른 기능을 충돌 없이 개발 가능
2. **환경 일관성**: “내 컴에서는 돼요”가 크게 줄어듦
3. **온보딩 속도**: 신규 개발자가 “며칠”이 아니라 “몇 시간” 안에 생산성 확보
4. **테스트 품질**: 서비스 간 통합 테스트를 쉽게 돌릴 수 있게 됨

### 배운 점

1. **인프라 복잡도를 과소평가하지 말 것**: “단순해 보이는 것”은 숨은 의존성이 많습니다
2. **마이그레이션은 철저히 검증할 것**: 스테이징에서 먼저 검증했어야 했습니다
3. **문서화는 생존 도구**: 내가 남긴 문서가 이 과정에서 여러 번 살려줬습니다
4. **점진적 변화가 낫다**: 처음에 한 번에 너무 많은 걸 바꾸려 했습니다

인프라 투자는 개발자 생산성으로 배당을 지급합니다. 그리고 그건 어떤 회사든 반드시 우선순위에 두어야 할 가치입니다.
