## Overview
- **목표**: 코드 커버리지 지표의 한계를 극복하고, 테스트 코드가 실제로 비즈니스 로직 오류를 잡아내는지 검증하기 위해 ‘Mutation Testing(돌연변이 테스트)’을 도입합니다.
- **범위**: 엔터프라이즈 오케스트레이터 프로젝트(`Ochestrator`)의 코어 모듈(프론트엔드 TypeScript + 백엔드 Python)
- **기대 효과**: 단순 라인 커버리지 대신 **Mutation Score**를 확보해 안정성과 신뢰도를 끌어올립니다.

우리는 보통 “커버리지가 높으면 안전하다”고 믿습니다. 하지만 이런 질문에는 답하기 어렵습니다.

**“테스트는 누가 테스트하지?”**

단순히 코드만 실행하고 제대로 된 assertion이 없는 테스트도 커버리지 지표에는 포함됩니다. 이 ‘커버리지 함정(coverage trap)’을 깨기 위해 돌연변이 테스트를 도입했습니다.

![Mutation Testing Flow](../assets/images/2026-02-01-18-mutation-flow.png)

## Implementation

### 1. TypeScript 환경: Stryker Mutator 도입

프론트엔드 및 공용 유틸을 포함한 TypeScript 환경에서는 [Stryker](https://stryker-mutator.io/)를 선택했습니다. Vitest와의 통합이 좋고, 설정이 비교적 단순했습니다.

- **기술 스택**: TypeScript, Vitest, Stryker Mutator
- **핵심 설정(`stryker.config.json`)**:
  ```json
  {
    "testRunner": "vitest",
    "reporters": ["html", "clear-text", "progress"],
    "concurrency": 4,
    "incremental": true,
    "mutate": [
      "src/utils/**/*.ts",
      "src/services/**/*.ts"
    ]
  }
  ```
  `incremental` 옵션을 켜서, 변경된 파일 중심으로 효율적으로 수행하도록 구성했습니다.

### 2. Python 환경: Cosmic Ray 도입

백엔드에서는 [Cosmic Ray](https://github.com/sixty-north/cosmic-ray)를 도입했습니다. Python의 동적 특성과 AST 조작을 활용해 강력한 돌연변이를 생성할 수 있습니다.

- **기술 스택**: Python, Pytest, Cosmic Ray, Docker
- **실행 아키텍처**: 돌연변이 테스트는 계산 비용이 크기 때문에, Docker로 여러 워커를 띄워 병렬 실행하도록 구성했습니다.
  ```yaml
  # Partial docker-compose.test.yaml
  cosmic-worker-1:
    command: uv run cosmic-ray worker cosmic.sqlite
  cosmic-runner:
    depends_on: [cosmic-worker-1, cosmic-worker-2]
    command: |
      uv run cosmic-ray init cosmic-ray.toml cosmic.sqlite
      uv run cosmic-ray exec cosmic-ray.toml cosmic.sqlite
  ```

## Debugging/Challenges

### 현실 사례: `VideoSplitter.ts`에서 살아남은 돌연변이들

가장 흥미로운 케이스는 비디오 분할 로직을 담당하는 [videoSplitter.ts](file:///d:/Coding/Company/Ochestrator/apps/frontend/src/utils/videoSplitter.ts)였습니다. 이 파일은 라인 커버리지가 95% 이상이었지만, Stryker의 결과는 충격적이었습니다.

- **문제**
  메모리 가용량을 체크하는 로직에서 다수의 돌연변이가 생존(survived)했습니다.
  ```typescript
  // Original Code
  if (availableMemory < requiredMemory) {
    throw new Error("Insufficient memory.");
  }
  ```
  Stryker가 이 코드를 `if (false)` 또는 `if (availableMemory <= requiredMemory)`로 바꿔도 기존 테스트는 모두 **통과**했습니다.

- **원인 분석**
  기존 테스트는 “에러가 나는지 여부”만 확인했고, 정확히 어떤 조건에서 에러가 나야 하는지에 대한 경계값 테스트가 없었습니다. 즉, 커버리지는 높았지만 로직 자체가 제대로 검증되지 않았던 것입니다.

- **해결**
  생존한 돌연변이를 ‘죽이기(kill)’ 위해 경계값 분석 기반 테스트를 추가했습니다.
  ```typescript
  test('Boundary value verification for memory', () => {
    // Simulate situations where memory is exactly equal to or slightly less than requiredMemory
    // ... reinforced test code ...
  });
  ```

## Results

- **성과**
  - 코어 유틸 모듈에서 **Survived Mutants** 12개를 발견하고 제거
  - 테스트를 단순한 “실행”에서 진짜 “검증”으로 끌어올림

- **핵심 지표**
  - **Mutation Score**: 초기 62% → 88%로 개선
  - **신뢰도**: 배포 전 `test:mutation` 스크립트를 돌려 회귀(regression) 위험을 낮춤

- **팀 피드백**
  - “이제 테스트를 믿고 리팩터링할 수 있게 됐다.”

## Key Takeaways

- **커버리지는 시작일 뿐**: 라인 커버리지는 ‘무엇이 테스트되지 않았는지’는 알려주지만, ‘테스트 품질’은 알려주지 않습니다.
- **돌연변이 테스트는 비싸지만 가치가 있다**: 실행 시간이 오래 걸릴 수 있지만(전체 실행은 수십 분), 코어 비즈니스 로직/복잡 유틸에는 투자 가치가 큽니다.
- **점진적 도입이 중요**: 전체에 한 번에 적용하기보다, `VideoSplitter` 같은 핵심 인프라 코드부터 성공 사례를 쌓아 확장하는 게 현실적입니다.
