## 소개

사이드 프로젝트에 **햄버거 메뉴, OAuth 로그인, 사용자별 설정** 같은 기능을 추가하면서, 순수 HTML/CSS/JavaScript(이하 *vanilla JS*)의 한계를 체감하기 시작했습니다. 상태를 가진 위젯이 늘어날수록 DOM 조작은 점점 “스파게티”가 되었고, 결국 이런 질문이 생겼습니다.

> “Svelte(또는 SvelteKit)를 도입해야 할까, 아니면 vanilla JS로 계속 밀어붙여야 할까?”

이 글은 프레임워크의 이점과 운영 리소스 제약을 함께 고려하기 위한 **실전 체크리스트**를 정리한 것입니다.

---

### 1. 나는 언제 프레임워크가 ‘진짜로’ 필요했나?

| 요구사항 | vanilla JS에서의 고통 | Svelte(Kit)의 장점 |
|-------------|----------------------|------------------------|
| 여러 컴포넌트 간 상태 공유 | 긴 `querySelector` 체인 + 즉흥 이벤트 버스 | `$:` 반응성, 전역 상태를 위한 Store |
| 클라이언트 라우팅(SPA 느낌) | History API 로직을 직접 구현해야 함 | 파일 기반 라우팅(Kit) |
| SEO + SSR | Django 템플릿은 SSR이지만 JS 위젯은 빈 `<div>`부터 시작 | SvelteKit SSR & prerender |
| 번들 최적화 | Webpack/Vite 튜닝을 직접 해야 함 | Vite 기반 빌드 + 기본 코드 스플리팅 |
| 팀/기능 확장 | 컨벤션 부재 → 온보딩 비용 ↑ | 컴포넌트/파일 컨벤션 내장 |

> **요약** — 공유 상태/재사용 컴포넌트/SEO 요구가 많아질수록 SvelteKit의 비용 대비 효과는 더 빨리 나타납니다.

---

### 2. vanilla로 계속 갈 때의 위험

1. **DOM 스파게티** — 누가 DOM을 바꾸는지 추적하기가 악몽이 됩니다.
2. **상태 불일치 버그** — 로그인/로그아웃, 다크 모드 토글 같은 게 쉽게 어긋납니다.
3. **테스트 비용 증가** — E2E는 장황한 DOM 셀렉터에 의존하게 됩니다.
4. **번들 피로도** — 페이지가 늘 때마다 캐싱/청크 분리 등을 수동 튜닝해야 합니다.

---

### 3. 옵션 비교

![Technology Stack Comparison](../assets/images/03/tech-stack-comparison.svg)

| 모델 | 장점 | 단점 |
|-------|------|------|
| **Vanilla JS** | Node 런타임이 없어 메모리 부담이 낮음 | 상태/라우팅을 직접 구현해야 함 |
| **Svelte 컴포넌트만 부분 도입** | 특정 위젯만 “끼워 넣기” 쉬움 | Django 뷰 간 이동은 전체 리로드 |
| **SvelteKit** | SPA 느낌 **+** SSR/SEO | 운영해야 할 Node 서버가 하나 더 생김 |

### 4. Python-first, 리소스 최소화 아키텍처

> 전제: SEO/GEO는 랜딩만 중요했고, Northflank 무료 티어를 쓰고 있었고, 운영은 Python 중심이 더 편했습니다.

1. **Backend** — Django(템플릿 + REST API)
2. **Frontend**
   * 랜딩: SEO 메타 태그가 있는 Django 템플릿
   * 대시보드/설정: 정적 에셋으로 제공되는 Svelte 번들
3. **빌드 파이프라인(멀티 스테이지 Docker)**
   ```bash
   FROM node:20 AS client-build
   WORKDIR /app
   COPY frontend/ .
   RUN npm ci ; npm run build

   FROM python:3.11-slim AS runtime
   WORKDIR /srv
   COPY --from=client-build /app/build/ /srv/static/
   COPY requirements.txt . ; pip install --no-cache-dir -r requirements.txt
   COPY . /srv
   CMD gunicorn config.wsgi --bind 0.0.0.0:$PORT
   ```
   → **런타임 이미지**에는 Node 바이너리가 포함되지 않아 RAM 사용량을 줄일 수 있습니다.

### 5. Cloudflare Worker BFF & 엣지 캐싱

> API 키를 브라우저에 노출하지 않으면서, Northflank 무료 쿼터를 아끼기 위해 API 앞단에 작은 **Cloudflare Worker**를 두었습니다.

![Cloudflare Worker BFF Sequence](../assets/images/03/cloudflare-worker-sequence.svg)

핵심 포인트:

* **보안** — API 키는 Worker의 secret에 저장되며, 브라우저로 내려가지 않습니다.
* **속도** — `cf: { cacheTtl: 60 }`로 JSON을 전 세계 엣지에서 캐싱합니다.
* **비용** — Workers Free 플랜으로 월 250만 요청까지 비용을 거의 0으로 억제할 수 있습니다.

```js
// cf-worker.js (snippet)
export default {
  async fetch(request, env) {
    if (request.method === 'OPTIONS') {
      return new Response(null, { status: 204, headers: { 'Access-Control-Allow-Origin': '*', ... } });
    }

    const api = new URL(env.UPSTREAM_URL);
    api.pathname = '/v1/orders';
    api.search = new URL(request.url).search;

    const upstream = await fetch(api, {
      headers: { 'X-API-KEY': env.API_KEY },
      cf: { cacheTtl: 60, cacheEverything: true }
    });

    const res = new Response(upstream.body, upstream);
    res.headers.set('Access-Control-Allow-Origin', '*');
    return res;
  }
};
```

### 6. Dev Container 워크플로우

> “내 컴에서는 되는데요?” 문제는 보통 **Docker 이미지 불일치**에서 시작합니다. 배포하는 이미지와 같은 컨테이너 안에서 개발하면 이런 버그를 크게 줄일 수 있습니다.

#### 왜 굳이?

* **환경 드리프트 0** — VS Code(또는 Codespaces)가 배포 런타임 이미지 안에서 실행되므로 Python/Node/OS 라이브러리가 운영과 일치합니다.
* **즉시 리로드** — 소스가 바인드 마운트되어 저장만 하면 Django autoreload/Vite HMR이 동작합니다.
* **디버깅 품질** — VS Code 디버거와 Svelte Inspector를 컨테이너 PID에 바로 붙일 수 있습니다.
* **온보딩 단축** — 새 기여자는 `Dev Containers: Reopen in Container`만 하면 빠르게 준비됩니다.

#### 최소 구성

1) `.devcontainer/devcontainer.json`
```jsonc
{
  "name": "myapp-dev",
  "dockerComposeFile": ["../docker-compose.dev.yml"],
  "service": "web",                  // Django container
  "workspaceFolder": "/srv",
  "shutdownAction": "stopCompose",
  "extensions": [
    "ms-python.python",
    "ms-python.vscode-pylance",
    "esbenp.prettier-vscode",
    "dbaeumer.vscode-eslint"
  ],
  "postCreateCommand": "pip install -r requirements.txt"
}
```

2) `docker-compose.dev.yml`
```yaml
version: "3.9"
services:
  web:
    build:
      context: .
      target: runtime          // reuse prod runtime stage
    command: python manage.py runserver 0.0.0.0:8000
    volumes:
      - ..:/srv                // live code mount
    ports:
      - "8000:8000"
    env_file: .env.dev

  client:
    build:
      context: ./client
      dockerfile: Dockerfile.dev
    command: npm run dev -- --host
    volumes:
      - ../client:/app
    ports:
      - "5173:5173"
```

3) `client/Dockerfile.dev`
```dockerfile
FROM node:20
WORKDIR /app
COPY package*.json ./
RUN npm install --legacy-peer-deps
CMD ["npm", "run", "dev", "--", "--host"]
```

#### 팁 & 주의사항

| 팁 | 이유 |
|-----|--------|
| `runtime` 스테이지 재사용 | Northflank 이미지와 100% 동일한 환경 보장 |
| `node_modules`는 마운트하지 않기 | 호스트-컨테이너 I/O 부담 완화 |
| 최초 오픈은 느릴 수 있음 | 이미지 pull 및 의존성 설치가 1회 수행됨 |
| `forwardPorts` 유지 | Codespaces에서 8000/5173 자동 노출 |

### 7. CI/CD & 배포 흐름

1. **로컬 커밋**
   ```bash
   git add . ; git commit -m "feat: landing SEO + CF worker" ; git push origin develop
   ```
2. **Northflank 파이프라인**
   * push 감지 → 멀티 스테이지 Dockerfile 빌드 → 컨테이너 배포
3. **(옵션) 수동 배포**
   ```bash
   docker build -t myapp:latest .
   docker push myrepo/myapp:latest ; nf deploy myapp
   ```
4. **Cloudflare Worker**
   ```bash
   wrangler publish --env production
   ```

### 8. 리소스 절감 레버

| 레버 | 효과 |
|-------|--------------|
| 런타임에서 Node 제거 | RAM 약 100MB 절감, 관리해야 할 CVE 감소 |
| JSON/정적 에셋 엣지 캐싱 | Django 요청 감소 → CPU 쿼터 절감 |
| LocMemCache → Redis(나중에) | 초반엔 싸게 시작하고 필요할 때만 확장 |
| 멀티 스테이지 Docker | 최종 이미지 < 200MB, 콜드 스타트 개선 |

---

### 9. 앞으로의 로드맵

* **UI 복잡도 증가** → `SvelteKit adapter-static`으로 전환(여전히 Node 서버 없이)
* **실시간 피드** → `Django Channels + Redis` 또는 SSE 추가
* **i18n/테마** → Svelte store 또는 `htmx` partial swap 활용
* **트래픽 급증** → Northflank 레플리카 확장 + Cloudflare TTL 튜닝

### 10. 의사결정 체크리스트(빠른 요약)

- [ ] 페이지가 10개를 넘길 가능성이 큰가?
- [ ] 인증/테마/알림 등 전역 상태를 공유해야 하는 지점이 2곳 이상인가?
- [ ] 대부분의 페이지가 SEO에 중요한가, 아니면 랜딩만 중요한가?
- [ ] Node 서버를 하나 더 운영/모니터링할 여력이 있는가?
- [ ] 팀에 JS 프레임워크 경험자가 있는가?

**3개 이상이 Yes**라면 SvelteKit을 고려할 만합니다. 그보다 적다면 *vanilla JS + 선택적 Svelte 위젯*이 더 합리적일 수 있습니다.

## 결론

프레임워크 도입은 **미래의 기술 부채에 대한 보험**에 가깝습니다. vanilla JS는 초반 속도를 올려주지만, 기능/팀/SEO 요구가 늘수록 **유지보수 비용이 급격히 상승**합니다.

오늘의 필요와 내일의 복잡도를 함께 보고, 마이그레이션의 **적절한 타이밍**을 선택하는 것이 핵심입니다. *Svelte 위젯 → SvelteKit* 같은 단계적 경로도 충분히 좋은 전략입니다.

> 💡 도구는 본질이 아닙니다. **지금 가장 빨리 가치를 전달**하면서도, **내일 스스로를 가두지 않는 선택**이 가장 중요합니다.
