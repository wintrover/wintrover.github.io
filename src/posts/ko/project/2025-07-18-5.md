## 소개

사이드 프로젝트를 하다가 문득 이런 생각이 들었습니다. “Google 로그인도 붙여볼까?”

_“Google 로그인? 버튼 하나 만들고 API 몇 개 호출하면 끝나는 거 아닌가?”_

그렇게 시작한 일이 생각보다 훨씬 긴 학습 여정이었습니다. OAuth 2.0이라는 단어를 처음 만난 순간부터, Google Cloud Console에서 길을 잃고, 환경변수를 실수로 Git에 올릴 뻔하고, 팝업이 왜 안 뜨는지 몰라 멘탈이 털리는 경험까지… 지금은 값진 자산이 됐습니다.

> “생각보다 훨씬 복잡했지만… 하나씩 해결해 나가면서 결국 해냈다!”

이 글은 주니어 개발자가 Google 로그인 구현에서 겪는 시행착오를 줄일 수 있도록, 제가 **Svelte/SvelteKit** 환경에서 실제로 구현하면서 선택한 방법과 맞닥뜨린 문제를 솔직하게 기록한 것입니다.

## 1. 시행착오 1 - Google Cloud Console 설정에서 헤매다

“Google 로그인 구현”을 검색하면 가장 먼저 나오는 단계가 **Google Cloud Console**에서 OAuth 2.0 Client ID를 만드는 것입니다. 그런데 막상 해보면 생각보다 단계가 많습니다.

### 실수 1: OAuth 동의 화면(Consent Screen)을 대충 설정함

동의 화면에서 “Internal”과 “External” 중 무엇을 선택해야 하는지부터 실수했습니다. 개인 프로젝트나 공개 서비스라면 **"External"**을 선택해야 합니다.

### 실수 2: Redirect URI를 잘못 설정함

가장 큰 함정은 **Authorized redirect URIs**였습니다. 포트 번호는 물론, 슬래시 하나까지 정확히 일치해야 `redirect_uri_mismatch`를 피할 수 있다는 걸 깨닫는 데에 몇 시간이 걸렸습니다.

초기(잘못된) 설정:
```
http://localhost:3000/callback
```

정상 설정:
```
http://localhost:5173          # Vite 개발 서버
https://cvfactory.dev          # 프로덕션 도메인
```

### 해결: 다시 차근차근 재설정하기

#### OAuth 플로우 선택 이유
PKCE를 포함한 Authorization Code Flow를 선택한 이유는 다음과 같습니다.
- **SPA 보안**: client secret을 프론트에 노출하지 않음
- **모바일 호환성**: 네이티브 앱과 동일한 플로우로 맞출 수 있음
- **Refresh Token 기반 장기 로그인**: (구성에 따라) 장기 인증이 가능

![Google OAuth Flow](../assets/images/05/google-oauth-flow.svg)

> **개발자 썰**
> “국제 스타트업 환경에서 OAuth를 구현할 때 가장 힘들었던 건 해외 서비스와의 연동이었어요. 로컬라이징이 제한된 공식 문서를 해석하는 게 제일 컸죠.”

헤맨 끝에 올바른 설정 방법을 정리했습니다.

#### Step 1: 프로젝트 생성 + API 활성화
- Google Cloud Console에서 프로젝트를 만들고 'Google Identity Services API'를 활성화합니다.

#### Step 2: OAuth 동의 화면 설정
```yaml
Application name: CV Factory
User support email: your-email@gmail.com
Authorized domains: cvfactory.dev (프로덕션)
```

#### Step 3: OAuth 2.0 Client ID 생성
```yaml
Application type: Web application
Name: Web Client
Authorized JavaScript origins:
  - http://localhost:5173
  - https://cvfactory.dev
Authorized redirect URIs:
  - http://localhost:5173
  - https://cvfactory.dev
```

설정이 끝나면 **Client ID**와 **Client Secret**을 받습니다. **중요**: Client Secret은 프론트엔드에서 사용하지 않습니다.

## 2. 시행착오 2 - 환경변수와 보안 실수

### 위험했던 순간: Client Secret을 프론트에 넣을 뻔
처음에는 Client ID와 Client Secret을 모두 프론트 코드에 넣어야 하는 줄 알았습니다.
```javascript
// ❌ 절대 하면 안 되는 코드
const GOOGLE_CLIENT_SECRET = "GOCSPX-very-secret-key"; // 위험!
```

다행히 커밋하기 전에, **PKCE를 사용하는 Authorization Code Flow에서는 프론트엔드에 Client Secret이 필요 없다는 것**을 알게 됐습니다.

### 실수 1: .env를 Git에 올릴 뻔
환경변수용 `.env` 파일을 만들고 습관적으로 `git add .`를 눌렀다가, `git reset HEAD .env`로 급히 취소했습니다. `.env.local`처럼 민감 정보를 담는 파일은 반드시 `.gitignore`에 포함해야 합니다.

### 실수 2: Vite에서 `VITE_` 프리픽스를 빠뜨림
Vite에서는 클라이언트에서 접근할 env 변수에 `VITE_` 접두사가 필요합니다. 이걸 몰라 `import.meta.env.GOOGLE_OAUTH_CLIENT_ID`가 계속 `undefined`였습니다.

#### 올바른 env 설정과 사용 예시
```bash
# .env.local (.gitignore에 포함되어야 함)
VITE_GOOGLE_OAUTH_CLIENT_ID=123456789-abc.apps.googleusercontent.com
```

```typescript
// lib/google-auth.ts
const CLIENT_ID = import.meta.env.VITE_GOOGLE_OAUTH_CLIENT_ID;

// env 누락에 대한 방어
if (!CLIENT_ID) {
  throw new Error("VITE_GOOGLE_OAUTH_CLIENT_ID is not set.");
}
```

## 3. 시행착오 3 - 구현과 디버깅 지옥

### 실수 1: Google API 스크립트 로딩 타이밍
SvelteKit에서는 컴포넌트가 마운트된 시점에 스크립트가 아직 로딩되지 않았을 수 있습니다. 그래서 **동적으로 스크립트를 로딩하는 방식**이 더 안정적이었습니다.
```typescript
// ✅ 동적 스크립트 로딩
private loadGoogleIdentityScript(): Promise<void> {
  return new Promise((resolve, reject) => {
    if (window.google?.accounts) {
      resolve();
      return;
    }
    const script = document.createElement('script');
    script.src = 'https://accounts.google.com/gsi/client';
    script.onload = () => resolve();
    script.onerror = () => reject(new Error('Failed to load Google Identity Services.'));
    document.head.appendChild(script);
  });
}
```

### 실수 2: 구 API(gapi.auth2)에 집착
초기에는 `gapi.auth2`를 쓰려 했는데, 이 API는 **deprecated**되었습니다. 2023년 이후에는 **Google Identity Services**를 사용하는 것이 권장됩니다.

### 실수 3: “토큰이 뭐지? 사용자 정보는 어떻게 가져오지?”
OAuth 흐름을 정확히 이해하지 못해, 토큰을 받은 뒤 무엇을 해야 할지 몰랐습니다. 나중에야 **access token으로 userinfo API를 호출해야 한다**는 걸 알게 됐습니다.
```typescript
// ✅ 토큰으로 사용자 정보 가져오기
// Test directly in editor: https://playground.example.com?code=...
callback: async (response: any) => {
  if (response.error) {
    reject(new Error("Google login failed."));
    return;
  }
  try {
    const userInfoResponse = await fetch(
      `https://www.googleapis.com/oauth2/v2/userinfo?access_token=${response.access_token}`
    );
    if (!userInfoResponse.ok) {
      throw new Error("Failed to fetch user information.");
    }
    const userInfo = await userInfoResponse.json();
    resolve(userInfo);
  } catch (error) {
    reject(new Error("Failed to fetch user information."));
  }
};
```
### 실수 4: 빈약한 에러 핸들링
처음엔 모든 에러를 `console.error`로만 처리했는데, 실제 사용자 경험에서는 **팝업 차단**, **팝업 닫힘**, **네트워크 오류** 같은 케이스별 안내가 중요했습니다.
```typescript
// ✅ 개선된 에러 처리
error_callback: (error: any) => {
  const errorMap = {
    'popup_closed': '로그인이 취소되었습니다',
    'popup_blocked': '팝업이 차단되었습니다. 브라우저 설정을 확인해주세요',
    'access_denied': '접근이 거부되었습니다',
    'invalid_request': '요청이 올바르지 않습니다',
    'unauthorized_client': '승인되지 않은 클라이언트입니다',
    'token_expired': '토큰이 만료되었습니다',
    'network_error': '네트워크 연결에 문제가 발생했습니다'
  };

  const errorMessage = errorMap[error.type] ||
                      `Unknown error: ${error.message}`;

  // 로깅/모니터링
  console.error(`[Google Auth Error] ${error.type}: ${error.message}`);
  sendToMonitoring(error);

  reject(new Error(errorMessage));
};
```

## 4. 시행착오 4 - UI/UX와 상태 관리

### 실수 1: 컴포넌트 내부 상태로만 관리
처음엔 컴포넌트 안에서만 상태를 관리했는데, 다른 컴포넌트가 로그인 상태를 알 수 없어 구조가 금방 무너졌습니다.

### 해결: Svelte Store로 전역 상태 관리
여러 컴포넌트에서 공유해야 하는 전역 상태(로그인 여부, 사용자 정보, 로딩 상태, 에러)를 **Svelte Store**로 올렸습니다.
```typescript
// lib/stores/auth.ts
import { writable } from "svelte/store";

function createAuthStore() {
  const { subscribe, set, update } = writable({
    isLoggedIn: false,
    user: null,
    isLoading: false,
    error: null,
  });

  return {
    subscribe,
    async login() {
      /* ... */
    },
    async logout() {
      /* ... */
    },
    async checkAuthStatus() {
      /* 페이지 로드 시 localStorage 확인 */
    },
  };
}

export const authStore = createAuthStore();
```

### GoogleAuthButton 컴포넌트 예시
로딩/에러/로그인 상태를 모두 처리하도록 구성했습니다.
```html
<!-- lib/components/GoogleAuthButton.svelte -->
<script lang="ts">
  import { authStore } from '../stores/auth.js';
  // ... (omitted) ...
</script>

<div class="auth-container">
  {#if $authStore.isLoggedIn && $authStore.user}
    <!-- 로그인 상태: 프로필 이미지, 이름, 드롭다운 메뉴 -->
  {:else}
    <!-- 로그아웃 상태: Google 로그인 버튼 -->
    <button class="login-button" on:click={() => authStore.login()} disabled={$authStore.isLoading}>
      {#if $authStore.isLoading}
        <div class="spinner"></div>
        <span>Logging in...</span>
      {:else}
        <span>Login with Google</span>
      {/if}
    </button>
  {/if}

  {#if $authStore.error}
    <div class="error-message">{$authStore.error}</div>
  {/if}
</div>

<style>
  .spinner {
    border: 2px solid #f3f3f3;
    border-top: 2px solid #3c4043;
    border-radius: 50%;
    width: 16px;
    height: 16px;
    animation: spin 1s linear infinite;
  }
  @keyframes spin { 0% { transform: rotate(0deg); } 100% { transform: rotate(360deg); } }
</style>
```
