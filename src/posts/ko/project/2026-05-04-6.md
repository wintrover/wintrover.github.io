## 1. 목표

마지막으로 남아 있던 bash wrapper (prek_hook_wrapper.sh)를 제거하고
Axiom의 모든 품질 게이트를 순수 Nim으로 전환하는 작업을 진행했다.

이 단계가 끝나면:

- 모든 gate는 Nim binary
- Git hook → Nim runner 직결
- shell 의존성 0

## 2. 문제 발생

마이그레이션 자체는 어렵지 않았다.

문제는 그 다음이었다.

```
gcc: fatal error: cannot execute 'cc1'
```

이상했다.

- gcc는 설치되어 있음
- 기존에는 잘 동작하던 코드
- 변경된 건 "gate 실행 방식"뿐

즉,

> 코드는 그대로인데 실행이 깨졌다

## 3. 잘못된 가설

처음엔 이렇게 생각했다.

- gcc 경로 문제인가?
- PATH가 잘못됐나?
- 특정 환경 변수 누락?

이 방향으로 가면 보통 이렇게 된다:

> 경로 하드코딩 → 임시 해결 → 나중에 또 깨짐

## 4. 실제 원인

문제는 훨씬 근본적이었다.

```nim
execCmdEx(cmd, env = customEnv)
```

이 한 줄.

Nim에서 env를 넘기면:

- 기존 환경을 "추가"하는 게 아니라
- **전체를 교체한다**

결과:

- PATH 사라짐
- HOME 사라짐
- toolchain 전부 접근 불가

즉,

> 우리는 환경을 수정한 게 아니라
> 프로세스를 격리시켜버린 것

## 5. 해결

해결 자체는 단순했다.

```nim
for (k, v) in envPairs():
  envTable[k] = v

for (k, v) in envVars:
  envTable[k] = v
```

부모 환경을 먼저 복원하고,
그 위에 필요한 값만 덮어쓴다.

## 6. 중요한 선택

여기서 갈림길이 있었다.

**선택 1**
- PATH만 복구
- 필요한 변수만 추가

**선택 2**
- 환경을 "완전히" 상속
- override만 적용

우리는 두 번째를 선택했다.

이유는 단순하다.

> 문제는 PATH가 아니라
> execution context 자체였기 때문

## 7. 결과

이 변경 이후:

- 모든 gate 정상 실행
- gcc / cc1 문제 사라짐
- 환경 관련 오류 재현 불가

그리고 더 중요한 변화:

- shell wrapper 완전 제거
- gate runner 단일화 (Nim)
- 실행 경로 100% 내부화

## 8. 회고

처음 목표는 단순했다.

> "bash를 제거하자"

하지만 실제로 한 일은 다르다.

**실행 권한을 회수한 것**

이전:
- 일부 실행은 shell에 위임
- 환경은 암묵적

이후:
- 모든 실행은 Nim 내부
- 환경은 명시적

## 9. 정리

이번 작업을 한 줄로 정리하면:

```
Shell-based execution → Deterministic runtime execution
```
