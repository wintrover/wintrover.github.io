## 핵심 질문

AI가 생성한 코드를 검증하는 엔진을 구축할 때, 아키텍트는 근본적인 딜레마에 직면한다. 규칙은 어디에 존재하며, 누가 이를 강제하는가?

Axiom의 답은 급진적이지만 필요하다: **규칙은 코드보다 먼저 정의되어야 한다.**

## Rule-First 원칙

전통적인 소프트웨어 개발은 암묵적인 패턴을 따른다:

```
1. 코드 작성
2. 테스트를 통한 버그 발견
3. 재발 방지 규칙 추가
4. 반복
```

이 반응형 접근 방식은 AI가 생성한 코드를 검증할 때 치명적으로 실패한다. LLM은 테스트를 통과하면서도 논리적 결함을 교묘히 은폐하는 '그럴듯한 오답'을 만드는 데 천재적이다. LLM 출력의 확률적 특성 때문에 버그는 미묘하게 다른 형태로 재등장할 수 있으며, 테스트 기반 탐지를 우회한다.규칙이 코드보다 먼저 선언되지 않으면, 검증 엔진은 AI가 만든 환각(Hallucination)에 맞춰 자신의 잣대를 깎아내리는—즉, 확률적 출력을 수용하기 위해 표준을 점진적으로 낮추는—치명적인 오류를 범하게 된다.

Axiom은 이 패러다임을 역전시킨다:

```
1. 규칙 선언 (불변량, 계약, 안전 속성)
2. 규칙을 검증 산출물로 인코딩
3. 코드가 규칙을 만족함을 증명
4. 그 후에만 실행 허용
```

이는 단순한 모범 사례가 아니다. 아키텍처적 필연이다. 코드 검사를 통해 자체 규칙을 발견하는 검증 엔진은 순환 논리에 빠진다: 동일한 코드에서 파생된 제약 조건에 대해 코드를 검증하며, 자기 정당화 루프를 생성한다.

### 실천에서의 의미

Axiom의 모든 변경은 다음 순서를 따른다:

1. **규칙 선언 우선**: 새로운 검증 기능을 추가할 때, 팀은 먼저 공식 규칙을 `CONTEXT.md` 또는 `docs/spec/`에 문서화한다
2. **규칙 대 테스트**: 테스트는 코드가 선언된 규칙을 올바르게 구현하는지 검증한다—코드가 독립적으로 작동하는지가 아니다
3. **구현 마지막**: 규칙과 테스트가 안정화된 후에만 구현이 시작된다

이것은 관료적 오버헤드가 아니다. 검증 엔진이 검증하는 코드와 동일한 엄격함을 준수하도록 보장하는 유일한 방법이다.

## 헌법-조례-실록 분리

Axiom은 규칙 계층을 세 개의 명확한 레이어로 조직하며, 수세기에 걸쳐 이 구조를 정제한 법률 시스템에서 차용했다.

| 레이어 | 키워드 | 파일 패턴 | 목적 | 변경 빈도 |
|--------|--------|-----------|------|-----------|
| **헌법** | **Context** | `CONTEXT.md` | 변하지 않는 "이유" | 희귀 (프로젝트 정체성 변경시에만) |
| **조례** | **Spec** | `docs/spec/*.md` | 상세 설계 명세 | 보통 (알고리즘/파라미터 변경) |
| **실록** | **History** | `docs/history/*.md` | 결정 기록과 맥락 | 빈번 (구현 로그) |

### 레이어 1: 헌법 (`Context`)

헌법은 다음을 포함한다:

- **프로젝트 목적**: 모든 결정을 안내하는 근본적인 "이유"
- **핵심 아키텍처 원칙**: Axiom의 정체성을 정의하는 불변 제약
- **신뢰 경계**: 엔진이 가정할 수 있는 것과 없는 것
- **품질 게이트**: 협상할 수 없는 통과/실패 기준

```markdown
## 2. 핵심 아키텍처 원칙 (SSOT)

### Purity Hierarchy

- **Core Calculation Tier**: `src/core` 계산은 오직 `func`를 사용
- **Boundary Executor Tier**: `proc`는 OS/인터페이스에서만 존재
- **Executor-only Rule**: 경계 `proc`는 비즈니스 로직을 소유할 수 없다
```

이 레이어는 드물게 변경된다—프로젝트 정체성 자체가 바뀔 때만. `CONTEXT.md`를 수정하는 것은 모든 하류 결정에 영향을 미치므로 아키텍처 검토가 필요하다.

### 레이어 2: 조례 (`Spec`)

조례는 상세 명세다:

- `bmc_core.md`: 모델 체킹 알고리즘, Z3 인터페이스, 타임아웃 규칙
- `uap_logic.md`: 범용 파이프라인 단계, 반례 승격, 불변량 정제
- `lean_proof.md`: Lean 4 의미론, tactic 전략, 재생 프로토콜

이 파일들은 필요 시 로드된다—방향은 `CONTEXT.md`에서 확인하고, 구현 시 특정 명세를 탐구한다. 이것은 헌법이 구현 세부 사항으로 혼잡해지는 것을 방지한다.

### 레이어 3: 실록 (`History`)

실록은 결정과 변경을 기록한다:

```markdown
## 2026-03-26
- 코어는 Bottom-up Sorter, Vault Manager, Axiom Orchestration Loop를 도입했다
- Stale axiom 차단과 Vault 기반 승격 경로를 확립했다
```

역사는 규칙과 분리된다. 규칙이 *왜* 존재하는지 이해해야 할 때 실록을 참조한다. 하지만 코드를 구현해야 할 때 헌법과 조례에 집중한다—역사적 노이즈를 헤매지 않고.

### 이 분리가 중요한 이유

이 구조가 없으면 검증 규칙이 구현 세부 사항과 역사적 우연에 얽힐 위험이 있다. 개발자는:

- "항상 이렇게 해왔다"(실록)를 "절대 변경하면 안 된다"(헌법)와 혼동할 수 있다
- 중요한 규칙을 변경 이력 항목에 묻어 코드 리뷰 중에 보이지 않게 할 수 있다
- 일시적 타협이 영구 아키텍처가 되도록 허용할 수 있다

Axiom의 삼중 레이어 분리는 다음을 보장한다:

- **원칙은 가시적으로 유지된다** (헌법)
- **메커니즘은 조직적으로 유지된다** (조례)
- **역사는 검색 가능하지만 분리된다** (실록)

## 무결성 하향 조정 금지: 정합성 강선

Axiom은 모든 규칙 중 하나를 가장 엄격하게 강제한다: **무결성은 증가만 가능하고 감소는 불가능하다.**

### func→proc 트랩

Nim의 타입 시스템은 두 가지 함수 선언을 허용한다:

```nim
# 순수: 부작용 없음, IO 없음, 외부 상태 변경 없음
func computeValue(x: int): int {.noSideEffect.} =
  result = x * 2

# 불순: IO 수행 가능, 상태 변경 가능, 예외 던지기 가능
proc processValue(x: int): int =
  echo "처리 중"  # 부작용: IO
  result = x * 2
```

Nim 2.0+에서는 `strictFuncs` 모드와 effect 추적이 이 구분을 더욱 강력하게 만든다:

```nim
# strictFuncs는 컴파일 타임에 순수성을 강제한다
{.push strictFuncs.}
func verifyLogic(node: Node): Result[void, Error] {.raises: [], tags: [].} =
  # 컴파일러는 외부 상태 변경이나 IO 호출을 거부한다
  if node.isValid: ok() else: err(InvalidNode)
{.pop.}
```

이것이 Nim이 Axiom 아키텍처에 최적화된 이유다. 언어 자체가 컴파일 타임에 순수성 경계를 강제한다—코드 리뷰에서가 아니라.

개발 중에 `func`를 `proc`로 변환하고 싶은 유혹이 있다:

> "여기에 빠른 디버그 로그를 추가해야 해. `func`를 `proc`로 변경하고, 로그를 추가하고, 나중에 다시 변경하겠다."

**이것은 Axiom에서 금지된다.**

왜? 왜냐하면:

1. **하향 조정은 일방문이다**: `proc`가 존재하면 개발자는 더 많은 부작용을 추가할 것이다. 변경이 확산된다.
2. **순수성은 이진이다**: 함수는 순수하거나 아니다. "약간 불순"은 무의미하다.
3. **검증은 순수성을 요구한다**: 예측할 수 없는 부작용이 있는 함수에 대해 수학적으로 속성을 증명할 수 없다.

### Axiom의 강제 메커니즘

Axiom은 **순수성 계층**을 유지한다:

```
┌─────────────────────────────────────────────────────────────┐
│ Core Calculation Tier (src/core)                            │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ func-only zone: 계산, 파싱, 정규화                      │ │
│ │ 반환: Result[T], Option[T], VerifiedType[T]            │ │
│ │ 금지: IO, 변경, 예외                                     │ │
│ └─────────────────────────────────────────────────────────┘ │
│                                                             │
│ Boundary Executor Tier (src/ui, src/cli, gateway)          │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ proc-allowed zone: OS, TTY, 파일시스템, 프로세스         │ │
│ │ 역할: 핵심 func 결과 실행, 의존성 주입                   │ │
│ │ 금지: 비즈니스 로직, 도메인 규칙                         │ │
│ └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

`src/core`에 `proc`가 나타나면 **설계 부채**로 표시된다—필수 리팩터링 계획이 있는 일시적 타협. 코드는 다음이 완료될 때까지 배포되지 않는다:

1. 실행자 경계가 명확히 표시된다
2. 백로그에 "func 승격" 작업이 존재한다
3. 불순성의 이유가 `CONTEXT.md`에 문서화된다

### 이 규칙을 위반하면 어떻게 되는가

Axiom에서 `func`를 `proc`로 하향 조정하는 것은 단순히 코드를 수정하는 행위가 아니다. 시스템의 수학적 무결성을 파괴하는 행위로 간주된다. 정당한 사유(`CONTEXT.md` 업데이트) 없는 하향 조정은 CI 단계에서 원천 차단된다.

Axiom의 CI는 **Procedure Gate**를 통해 이를 강제한다:

1. `src/core` 파일이 변경되면 `proc` 추가 확인
2. diff에 `func`→`proc` 변환이 나타나면, **커밋 실패**
3. 해당 문서 없이 `proc`가 존재하면, **커밋 실패**

유일한 예외: `--style-only` 변경(오타, 주석, 공백). 그 외 모든 것은 아키텍처적 정당화가 필요하다.

컴파일 타임에 Nim의 effect 시스템은 추가 강제를 제공한다:

```nim
# [Layer 1: Core Calculation Tier]{.push strictFuncs.}
func verifyNodeIntegrity(node: AxiomNode): Result[void, IntegrityError] {.raises: [].} =
  ## IO 없음, 변경 없음, 예외 없음 - 수학적으로 순수
  if node.fingerprint == node.computedFingerprint():
    ok()
  else:
    err(IntegrityError.FingerprintMismatch)

# [Layer 2: Boundary Executor Tier]
proc executeVerification(node: AxiomNode, output: File) {.raises: [IOError].} =
  ## 얇은 실행자: 의존성을 주입하고 모든 로직은 func에 위임
  let result = verifyNodeIntegrity(node)  # 순수 함수 호출
  if result.isOk:
    output.write("검증 통과\n")  # IO는 경계에 격리
  else:
    output.write("검증 실패: " & $result.error & "\n")
{.pop.}
```

패턴에 주목하라: `verifyNodeIntegrity`는 순수 수학이다. `executeVerification`은 IO를 처리하는얇은 래퍼다. 비즈니스 로직은 절대 불순 영역으로 넘어가지 않는다.

### 실제 예시

리포트 렌더링을 구현할 때 Axiom은 선택에 직면했다:

```nim
# 처음에 유혹받는 접근:
proc renderReport(data: seq[ProofResult]) =
  # 직접 IO: stdout에 기록
  for r in data:
    echo r.format()

# 강제된 순수성 접근:
func renderReportLines(data: seq[ProofResult]): seq[string] {.noSideEffect.} =
  # 순수 계산: 문자열 반환
  for r in data:
    result.add r.format()

proc writeReport(data: seq[ProofResults]) =
  # 얇은 실행자: 출력 스트림 주입
  let lines = renderReportLines(data)
  for line in lines:
    echo line
```

두 번째 접근은 관심사를 분리한다:

- `renderReportLines`는 순수하고, 테스트 가능하며, 검증 가능하다
- `writeReport`는 파일 출력, 네트워크 출력, 또는 테스트 목으로 교체할 수 있는 간단한 실행자다

이것은 과도한 엔지니어링이 아니다. **검증을 가능하게 만드는 것이다.**

## 이 원칙들이 존재하는 이유

Axiom은 AI가 생성한 코드를 검증한다. 이를 신뢰성 있게 수행하려면 검증하는 코드보다 **더 엄격해야 한다**.

Rule-First 설계는 다음을 보장한다:

- 검증 규칙은 기존 코드를 정당화하기 위해 후속 조정되지 않는다
- 아키텍처 원칙은 명시적이고, 발견 가능하며, 강제된다
- 무결성은 한 방향으로만 이동한다: 상향으로

헌법-조례-실록 분리는 다음을 보장한다:

- 원칙은 세부 사항에 잠기지 않고 가시적으로 유지된다
- 구현자는 명세를 쉽게 찾는다
- 역사는 정보를 제공하지만 혼잡하게 하지 않는다

무결성 하향 조정 금지 정책은 다음을 보장한다:

- 순수성은 편의를 위해 희생되지 않는다
- 검증은 수학적으로 건전하게 유지된다
- 기술 부채는 가시적이고, 추적되며, 일시적이다

## 결론

이 원칙들은 열망적이지 않다—CI 게이트, 코드 리뷰, 아키텍처 문서에 의해 강제된다. Axiom의 모든 커밋은 Procedure Gate를 통과해야 한다. 핵심의 모든 `proc`는 존재를 정당화해야 한다. 모든 규칙 변경은 구현 전에 문서화되어야 한다.

> **아키텍처 자체가 검증 가능해야 한다.**

이것이 검증 엔진을 구축한다는 의미다: **확률적 코드를 확률적 도구로 검증할 수 없다.** 기반은 결정론적이고, 수학적으로 증명되며, 아키텍처적으로 순수해야 한다.

다음 포스트에서는 Axiom이 경계 모델 체킹을 구현하는 방법을 탐구할 것이다—수학적 약속을 실행 가능한 증명으로 변환하는 수학적 기초.
