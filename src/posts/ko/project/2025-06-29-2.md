## 소개

지난 글에 이어 이번 글에서는 **CVFactory** 백엔드의 내부를 공개합니다. 레포에서 `@/Backend` 디렉토리에 해당하는 부분입니다.

겉보기에는 작은 서비스지만, 실제로는 꽤 많은 구성 요소가 함께 움직입니다.

* 얇은 HTTP 인터페이스를 위한 **FastAPI**
* 비동기/내고장성 작업 큐를 위한 **Celery**
* 로그인 뒤에 숨어 있는 채용 공고 텍스트를 긁어오기 위한 **Playwright**
* 프롬프트 템플릿과 LLM 오케스트레이션을 위한 **LangChain**
* 모든 걸 재현 가능한 단일 컨테이너로 묶기 위한 **Docker + Supervisord**

이 글의 목적은 설계 선택의 이유, 코드 스니펫, 그리고 실전에서 겪는 함정들을 공유해 여러분이 같은 패턴을 자신의 프로젝트에 재사용/확장할 수 있게 하는 것입니다.

### 1. 왜 Django에서 백엔드를 분리했나?

Django는 세션 기반 웹 앱에 강하지만, LLM 호출이나 헤드리스 브라우저 자동화처럼 오래 걸리는 작업은 asyncio 루프를 막거나 Gunicorn 워커를 소진시킬 수 있습니다. 이런 작업을 **전용 FastAPI + Celery 스택**으로 오프로딩하면, 메인 웹 앱은 더 빠르게 반응하고 수평 확장도 쉬워집니다.

### 2. 태스크 파이프라인 자세히 보기

1. **`extract_html.py`** — URL을 입력받아 Playwright 컨텍스트를 띄우고, 필요하면 인증한 뒤 원본 HTML을 추출합니다.
2. **`text_extraction.py`** — `BeautifulSoup`로 HTML을 정리하고 네비게이션 같은 보일러플레이트를 제거합니다.
3. **`content_filtering.py`** — 비속어 필터를 적용하고 PII(개인식별정보)를 마스킹합니다.
4. **`cover_letter_generation.py`** — 프롬프트를 만들고 LLM을 호출한 뒤 결과를 클라이언트로 스트리밍합니다.

각 단계는 **멱등성(idempotent)**을 갖도록 만들었고, `/Backend/logs`에 로깅해 재실행 시 같은 페이지를 불필요하게 다시 크롤링하지 않도록 했습니다.

```python
# tasks/cover_letter_generation.py (snippet)
from celery import shared_task
from utils.logging_utils import task_logger
from utils.celery_utils import get_openai_client

@shared_task(bind=True, acks_late=True, autoretry_for=(Exception,), retry_backoff=True)
def generate_cover_letter(self, clean_text: str, profile: dict) -> str:
    """Generates a cover-letter draft from sanitized JD + user profile."""
    task_logger.info("Starting generation task", extra={"task_id": self.request.id})

    client = get_openai_client()
    prompt = _build_prompt(clean_text, profile)
    response = client.chat.completions.create(model="gpt-4o", messages=prompt)

    task_logger.info("LLM finished", extra={"usage": response.usage})
    return response.choices[0].message.content
```

### 3. 견고한 로깅 & 에러 핸들링

모든 함수는 **구조화 로깅(structured logging)**과 **세분화된 예외 처리**로 감싸 두었습니다. Playwright에서 에러가 나더라도 워커 전체가 무너지는 일이 없도록 하는 것이 핵심입니다. 운영 환경에서는 로그를 CloudWatch로 보내고, 로컬에서는 파일로 저장합니다.

* `retry_backoff=True`를 사용해 일시적인 오류에 대해 지수 백오프 재시도를 적용합니다.
* 전체 트레이스백은 캡처하되, 로그 전송 전에는 민감한 환경변수는 마스킹합니다.

### 4. 로컬 개발은 한 줄로

```bash
docker compose -f docker-compose.yml up --build --remove-orphans
```

### 5. 배운 점

* **태스크는 작고 직렬화 가능하게** — Celery에는 JSON 직렬화 가능한 페이로드만 넘기는 편이 안전합니다.
* **웹 워커에서 스크래핑하지 않기** — I/O가 무거운 스크래핑은 전용 워커로 넘겨 타임아웃과 병목을 피합니다.
* **타입이 있는 설정을 활용하기** — `core/config.py`에서 `pydantic.BaseSettings`를 쓰면 잘못된 env 설정을 시작 단계에서 잡을 수 있습니다.

### 마무리

백엔드는 사용자 눈에 잘 보이지 않지만, 프론트엔드에서 “AI가 마법처럼 동작하는 경험”을 가능하게 하는 핵심입니다. HTTP I/O, 작업 큐, 스크래핑, LLM 호출을 역할별로 모듈화하면 관측 가능하고 확장 가능하며 변경하기 쉬운 파이프라인을 얻을 수 있습니다.
